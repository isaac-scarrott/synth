# 2026-07-06

## 2026-07-06 — Row ⋯ frames carry the branch's session creates (both designs + native app)

A row's ⋯ kebab drills the ⌘K palette to that row's frame, but the workspace and session frames
offered no way to start a session — a workspace's dots showed only worktree/settings/rename/remove,
and a session's dots only rename/delete. ⌘K root already answers "what's relevant here" with its
context actions: the context branch's creates (New terminal / New Claude Code / New browser) appear
even when the anchor is a workspace or session. The kebab frames now match that scoping.

- **Workspace frame:** after "New worktree…", the three session creates targeting the workspace's
  active branch (`.branch--active`, else its first branch — the same fallback ⌘K root's
  `contextBranch` uses). A ctx chip names the branch, since the crumb only establishes the workspace.
- **Session frame:** the three creates as *siblings* on the session's own branch (mirroring the `a`
  shortcut, which adds a sibling session), ahead of Rename / Delete, ctx chip naming the branch.
- **Branch frame:** unchanged — it already led with the three creates, bare-labelled (its crumb is
  the branch). In the app the three items are now built by one shared `sessionCreates(in:ctx:)`
  helper used by all three frames.
- **Deliberately not everything:** the kebab frame stays a scoped slice of ⌘K — no global actions
  (toggle sidebar, settings, new workspace), which remain root-only.

Landed in `working.html` + `big-picture-design.html` (invariant checked) and the native app
(`Palette.swift` — `workspaceFrame`, `sessionFrame`, `branchFrame`).

## 2026-07-06 — Terminal "finished" joins the notification deck; cards widen to 320px (both designs + native app)

A background terminal's per-command lifecycle already reached the notification seam (the injected
zsh reporter → `term-run`/`term-idle`/`term-error` → `routeTransition`), but only error and
needs-input ever surfaced as toasts — a long command *finishing* off-screen was ambient-only
(unread bullet + row pulse), which read as "notifications don't work for terminals". Now the full
terminal lifecycle notifies.

- **Done toast (terminal/browser only).** A non-Claude background session settling live→idle raises
  a green-✓ glass toast ("finished") in the deck, ranked behind error and needs-input
  (error > input > done, newest within a kind). It asks for nothing, so it never persists: it
  dismisses itself after 6s (seq-guarded in the app so the timer can't kill a newer toast; a
  standing card whose kind changes is rebuilt, not restyled). The ambient layer (unread + row
  pulse) still fires alongside.
- **Claude's done stays ambient** — no toast, unchanged from the 2026-07-05 decision: its finish
  already reads in the chat, and the needs-input race fix leans on done being quiet. The unfocused
  path is also unchanged (Notification Center already bannered done for every kind; it now shares
  `notifVerb` for its title).
- **Cards widen 292 → 320px** in both designs (`.notifs`) and the app (`NotificationDeck.cardWidth`,
  now the single width constant).
- **Harness seams added while verifying:** `automation.notifs` (ControlServer, SYNTH_AUTOMATION=1)
  returns the deck exactly as `store.notifOrder` renders it plus `NSApp.isActive`, so a headless
  run can assert standing toasts without pixels; `SYNTH_STATE_DIR` overrides the persistence dir —
  required because Application Support resolves the *real* user home regardless of `$HOME`, so a
  driven test instance would otherwise share (last-writer-wins) the user's live state.json.
- **Verified live.** HTML: driven in a browser — deck of four ordered error > input ×2 > done, done
  auto-dismissed at 6s, cards measure 320px. App: an isolated instance (SYNTH_STATE_DIR) fed real
  wire-format lines over the hook socket — `term-error` raised a persistent error toast,
  `term-run`→`term-idle` raised the done toast behind it and auto-dismissed at 6s, Claude
  `working`→`idle` raised no toast; `swift build` clean. Both designs byte-identical except title +
  the browser/simulator demo rows.

## 2026-07-06 — Every session kind auto-names its row (and the child-session env fix that unblocked Claude's)

A row's name should tell you what's happening inside it without opening it. Claude rows already had
the pipeline (Claude Code's evolving `ai-title`, read from the transcript by `synth-hook`, forwarded
over the hook socket as a `title` line); terminals and browsers now feed the same seam, and a bug
that silently starved the Claude path is fixed. Hand-renamed rows stay frozen everywhere —
`titleIsCustom` gates all three sources at the single `.titleChanged` reducer.

- **Fix: spawned claudes were "child sessions", so ai-titles never existed.** When Synth itself is
  launched from inside a Claude Code session (dev.sh in a claude turn), the parent's env markers
  (`CLAUDECODE`, `CLAUDE_CODE_CHILD_SESSION`, `CLAUDE_CODE_SESSION_ID`, …) leaked into every PTY.
  Claude Code treats such sessions as nested children: no transcript persisted to
  `~/.claude/projects/`, no `ai-title` generated — `readAITitle` always found nothing and rows sat
  at "Claude Code" forever. `HookEnvironment.decorate` now scrubs the markers, so sessions inside
  Synth are always top-level. (Diagnosed live: app-side rename plumbing proved healthy by socket
  injection; live sessions had `CLAUDE_CODE_CHILD_SESSION=1` and no transcript on disk.)
- **Terminal rows auto-name after the running command.** The injected zsh preexec already armed a
  0.5s timer before signalling `term-run` (so `ls`-class commands never strobe); the typed command
  line now rides along — `synth-hook report --signal term-run --title <cmd>` — and becomes the row
  title (first line, whitespace collapsed, capped at 60 chars). Same gate as the green dot: only a
  command that outlives the timer renames the row, and the name persists after it exits (the row
  reads as "the `pnpm dev` terminal", like Claude's title persists between turns). `claude` remains
  excluded — its richer pipeline owns those rows.
- **Browser rows auto-name after the page title, falling back to the URL.** `.browserNavigated`
  already set host+path (`localhost:8733/palette`); `.browserPageTitled` — which previously only
  labelled the Recents list — now also renames the row, so the fallback stands exactly until the
  page reports a real title (SPA route changes included, via the engine's title-change callback).

Native app only (`Hooks.swift`, `synth-hook/main.swift`, `Store.swift`) — the designs don't
simulate auto-naming. Restart Synth to pick it up: the env scrub applies to newly decorated PTYs,
and the zsh reporter is rewritten into the injected ZDOTDIR at app launch.

## Sidebar indicators: soft glow + entry pop, one shared axis (both designs + native app)

The liveness dots dropped the two-tier ring (solid dot + hard 2.5px `box-shadow` ring) for a soft
round glow — two blurred colour-matched layers (`0 0 3px @ .55` + `0 0 8px 2px @ ~.3`); the chat
`work-dot` matches. Every `.ind` slot now animates in with `ind-in` (240ms
`cubic-bezier(0.34,1.56,0.64,1)`, opacity 0 / scale 0.4 → rest), retriggering whenever an indicator
appears or swaps state; `attn-breathe` moved from `.ind--input` onto its inner svg so the loop
composes with the slot's entry pop. Reduced motion disables the entry.

Native port (`Sidebar.swift`): `Dot` lost `halo`/`haloOpacity` and draws the glow with two stacked
`.shadow`s; a new `Ind` view is the fixed 16×16 slot with the spring entry
(`.spring(response: 0.24, dampingFraction: 0.6)`), skipped under `accessibilityReduceMotion`. The
port also fixed the indicator/⋯ alignment the two-tier era hid: `BranchRollup` right-flushed its
dot in a `minWidth: 16` frame (5pt off the session axis) and the workspace head had no slot at all
— all three levels now share `Ind`, so indicators and kebabs each form one straight column down the
sidebar, and idle sessions keep an empty slot so hover swaps never shift layout.

## Claude's done now raises a transient toast (both designs + native app)

A background Claude session settling to idle no longer stays ambient-only — it raises the same
transient green-check done toast as a terminal/browser (6s, self-dismissing), on top of the
existing unread bullet and row pulse. The "its finish already reads in the chat" rationale didn't
hold off-screen: with the session backgrounded the chat isn't visible, so a finished Claude turn
had no focused-window surface at all (the unfocused path already posted a "Claude finished"
Notification Center banner — the two surfaces now agree).

- **Design:** `notifyOnTransition` drops the `data-type !== 'ai'` guard; every live→idle
  background transition calls `notify(el, 'done')` (which also sets unread). `notifWhat` already
  phrased it — "Claude finished".
- **Native:** `routeTransition`'s focused idle branch drops the `.claudeCode` special case and
  always calls `raiseInApp(id, .done)`; the NC path is unchanged.

## 2026-07-06 — Browser rows carry no status indicator (both designs + native app)

The green "running" dot on browser rows said nothing: it lit when the engine mounted and never
changed again, so it wasn't liveness — it was furniture. Browser sessions now carry no status at
all: the row's indicator slot stays empty for life (the slot itself remains, holding the shared
right axis), and since the row never transitions it never raises a done toast or roll-up state.
Terminals, Claude, and the simulator keep their indicators unchanged.

- **Designs:** the big-picture browser demo row drops its `sdot--run`; `SESSION_KINDS.browser`
  spawns at `idle` (empty slot); the ⌥N notif demo skips browser rows so the demo can't paint a
  state onto one. Subset invariant re-verified (diff = title + browser/simulator demo rows).
- **Native:** `newBrowser` spawns `.idle` instead of `.running`, and `BrowserManager` no longer
  posts `.statusChanged(.running)` when the engine mounts. That post doubled as the re-render
  nudge for a pane that saw nil during CEF's reentrant bootstrap — replaced by an observable
  `generation` counter on `BrowserManager` (bumped when a controller is cached; `controller(for:)`
  reads it, subscribing any calling render). Status never persisted, so restored rows need no
  migration. `notifVerb`'s browser cases stay for switch exhaustiveness only.

## 2026-07-06 — `d` deletes through the ⌘K confirm frame (both designs + native app)

Pressing `d` on the focused sidebar row now drops into the ⌘K palette at that row's delete-confirm
frame — the same confirmFrame the kebab and palette flows already use — instead of raising the
row's inline confirm popover. One confirm surface everywhere: same Delete/Remove verbs, the
worktree remove-vs-delete fork, ↵ confirm · esc cancel. The inline popover machinery
(openMenu/showConfirm in the designs, ActiveMenu/menuConfirming in the app) is now unreachable —
left in place for now.

- **Designs:** `requestDelete` → `openPaletteAt(confirmFrame(row))`.
- **Native:** `Store.requestDelete` builds the matching palette confirm frame (workspace / branch /
  session) over a root frame. New SYNTH_AUTOMATION control verbs (`automation.requestDelete`,
  `.paletteMove`, `.paletteEnter`, `.palette`) drive and inspect the flow on machines where TCC
  blocks synthetic keys — verified end-to-end: confirm frame opens (legacy menu stays closed),
  Cancel preserves the session, ↵ deletes and persists.
