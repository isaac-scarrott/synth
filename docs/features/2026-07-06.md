## 2026-07-06 — Row ⋯ frames carry the branch's session creates (both designs + native app)

A row's ⋯ kebab drills the ⌘K palette to that row's frame, but the workspace and session frames
offered no way to start a session — a workspace's dots showed only worktree/settings/rename/remove,
and a session's dots only rename/delete. ⌘K root already answers "what's relevant here" with its
context actions: the context branch's creates (New terminal / New Claude Code / New browser) appear
even when the anchor is a workspace or session. The kebab frames now match that scoping.

- **Workspace frame:** after "New worktree…", the three session creates targeting the workspace's
  active branch (`.branch--active`, else its first branch — the same fallback ⌘K root's
  `contextBranch` uses). A ctx chip names the branch, since the crumb only establishes the workspace.
- **Session frame:** the three creates as *siblings* on the session's own branch (mirroring the `a`
  shortcut, which adds a sibling session), ahead of Rename / Delete, ctx chip naming the branch.
- **Branch frame:** unchanged — it already led with the three creates, bare-labelled (its crumb is
  the branch). In the app the three items are now built by one shared `sessionCreates(in:ctx:)`
  helper used by all three frames.
- **Deliberately not everything:** the kebab frame stays a scoped slice of ⌘K — no global actions
  (toggle sidebar, settings, new workspace), which remain root-only.

Landed in `working.html` + `big-picture-design.html` (invariant checked) and the native app
(`Palette.swift` — `workspaceFrame`, `sessionFrame`, `branchFrame`).

## 2026-07-06 — Every session kind auto-names its row (and the child-session env fix that unblocked Claude's)

A row's name should tell you what's happening inside it without opening it. Claude rows already had
the pipeline (Claude Code's evolving `ai-title`, read from the transcript by `synth-hook`, forwarded
over the hook socket as a `title` line); terminals and browsers now feed the same seam, and a bug
that silently starved the Claude path is fixed. Hand-renamed rows stay frozen everywhere —
`titleIsCustom` gates all three sources at the single `.titleChanged` reducer.

- **Fix: spawned claudes were "child sessions", so ai-titles never existed.** When Synth itself is
  launched from inside a Claude Code session (dev.sh in a claude turn), the parent's env markers
  (`CLAUDECODE`, `CLAUDE_CODE_CHILD_SESSION`, `CLAUDE_CODE_SESSION_ID`, …) leaked into every PTY.
  Claude Code treats such sessions as nested children: no transcript persisted to
  `~/.claude/projects/`, no `ai-title` generated — `readAITitle` always found nothing and rows sat
  at "Claude Code" forever. `HookEnvironment.decorate` now scrubs the markers, so sessions inside
  Synth are always top-level. (Diagnosed live: app-side rename plumbing proved healthy by socket
  injection; live sessions had `CLAUDE_CODE_CHILD_SESSION=1` and no transcript on disk.)
- **Terminal rows auto-name after the running command.** The injected zsh preexec already armed a
  0.5s timer before signalling `term-run` (so `ls`-class commands never strobe); the typed command
  line now rides along — `synth-hook report --signal term-run --title <cmd>` — and becomes the row
  title (first line, whitespace collapsed, capped at 60 chars). Same gate as the green dot: only a
  command that outlives the timer renames the row, and the name persists after it exits (the row
  reads as "the `pnpm dev` terminal", like Claude's title persists between turns). `claude` remains
  excluded — its richer pipeline owns those rows.
- **Browser rows auto-name after the page title, falling back to the URL.** `.browserNavigated`
  already set host+path (`localhost:8733/palette`); `.browserPageTitled` — which previously only
  labelled the Recents list — now also renames the row, so the fallback stands exactly until the
  page reports a real title (SPA route changes included, via the engine's title-change callback).

Native app only (`Hooks.swift`, `synth-hook/main.swift`, `Store.swift`) — the designs don't
simulate auto-naming. Restart Synth to pick it up: the env scrub applies to newly decorated PTYs,
and the zsh reporter is rewritten into the injected ZDOTDIR at app launch.

## 2026-07-06 — A clean exit closes its session; the done toast outlives the row (native app)

A session whose child exits cleanly now closes itself — `exit` in a terminal, or quitting a
Claude session — instead of lingering as a dead "exited" row the user has to delete by hand.
The done toast still fires first (it self-dismisses, so it can safely outlive its session):
`InAppNotif` snapshots kind/title/workspace-color at raise time, and `notifOrder` keeps an
exit-close toast whose session has vanished — every other toast still drops with its row. A
nonzero exit keeps the row, showing the error: failures should be seen and inspected, not
vanish with the process.

Supporting changes that make the exit signal honest for Claude sessions:

- The login shell now **`exec`s `claude`** (was: runs it, dropping back to the shell after).
  The session exists to run claude, so claude's end *is* the PTY child exiting — the same
  child-exited signal a terminal's `exit` raises.
- `Session.spawnedKind` records the creation label. A session spawned as Claude never reverts
  to a plain terminal on claude-end: with exec there is no shell to fall back to, so the
  claude-end is either the process about to exit or a `/clear` end/start pair — neither
  should blip the kind.

**The exit code can't ride the PTY** — verified empirically: libghostty wraps every macOS PTY
child in `/usr/bin/login`, and login exits 0 whatever its child's status was (`login … -c
"exit 5"` → 0), so ghostty's child-exited action always reports 0 and the clean-vs-failure
fork would otherwise be dead code. The true status travels over the hook socket instead —
the seam that exists for exactly this kind of derived fact:

- **Plain terminals:** the injected zshrc gains a `zshexit` hook — `synth-hook report --exit
  $?` (foreground, the shell is about to die).
- **Claude sessions:** the shim's launch role no longer execs the real claude; it
  posix_spawns it (shim ignores INT/QUIT, child keeps defaults), waits, reports the code
  over the socket, and exits with the same code.
- **Store:** `.exitCodeReported` stashes the code per session; the `.exited` that follows
  moments later prefers it over the PTY's 0. The user-interrupt statuses 130 (SIGINT) and
  143 (SIGTERM) close clean — the same neutrality the per-command reporter applies, so a
  Ctrl-C'd process doesn't die as an error row. `exit` with no args exits with the last
  command's status (zsh semantics), so an `exit` right after a failed command keeps an
  error row — accepted.

Native-only: the design mocks have no real PTY children to exit. Verified end-to-end in the
isolated harness (`SYNTH_STATE_DIR` + `SYNTH_AUTOMATION=1`, driven over the control socket):
clean background exit → row closes + done toast from its snapshot, standing ~6s after the
row is gone; clean open-session exit → row closes, no toast, cursor falls to the branch;
`claude <bad-flag>` (exit 1) → row stays, status error; zshexit reports the real code (7)
over the socket. A new read-only `automation.nav` control verb (rows / cursor / open
session) makes row lifecycle and cursor fallback assertable without pixels — this machine's
TCC blocks both synthetic keys and window capture.

## 2026-07-06 — Browser session, stage four decided: a browser can belong to a Claude session (ADR-0011 amended)

Grilled to a decision set. A browser session can be **contained** by a Claude Code session in the same
branch — true containment, not just a routing pointer — while user-created browsers stay branch-tier
siblings. Slice A (design pass) lands with this entry; slice B is the native port, gated separately.

- **Semantic: true containment, shared-visible.** An owned browser nests under its Claude row and
  lives/dies with it, but stays on the shared surface: `browser_list` returns every browser in the
  worktree (owned rows annotated with their owner), any claude can drive any browser, and the user can
  always take the wheel. Ownership is exclusive only for nesting, cascade, and comment routing — the
  no-headless-twin ethos is untouched.
- **Establishment: create-only + manual re-parent.** `browser_create` stamps the calling claude as
  owner (the launch shim exports the claude session id into claude's env; the MCP server inherits it
  and passes it over the control socket). ⌘K browsers are born unowned. Driving a browser never adopts
  it — structure only changes on creation or explicit user action. Row kebab gains "Move under
  <claude>…" and "Detach".
- **Lifecycle: cascade with confirmation.** Deleting an owning Claude row closes its browsers; the
  confirm names them ("also closes 2 browsers"); Detach beforehand is the escape hatch. Ownership keys
  off the Synth row UUID, so it survives claude exits and `--resume`.
- **Sidebar: nested, same session styling.** Owned browsers render one indent step under their owner
  with the existing session-tier dials — no caret, no collapse state, no fifth visual register. Amends
  the locked three-tier hierarchy to "three tiers + a containment indent".
- **Comment ladder: owner → boot owner → spawn.** Owner live → deliver. Owner dormant → the existing
  boot-and-wait path. Unowned → **always spawn a fresh claude** (replaces the most-active-in-branch
  fallback for unowned browsers), and **the spawned claude adopts the browser** — creation-time
  stamping, so the next comment hits rung one. The hook-liveness security boundary is unchanged:
  submit only after confirmed liveness, including for the freshly spawned session.
- **Spawn moment: silent, stay in browser.** No confirmation — the bar chip reads "→ New Claude
  session" before send, so nothing is hidden. The claude row appears (with the browser re-nested under
  it), boots in the background, receives the standard located-context composition as its first
  message, and takes its ai-title from there. Focus stays on the page.
- **Chosen knowingly:** with an active claude already on the branch, commenting from a fresh ⌘K
  browser spawns a stranger rather than routing to the active one — re-parent first if that's what you
  want. And a ⌘K browser adopted by a spawned claude cascade-deletes with that claude even though the
  browser predates it — the confirm dialog is the guard.
- **External claudes** (running outside Synth in a managed worktree) have no Synth row to own, so
  their `browser_create` yields an unowned sibling. A detached browser is fully unowned again — its
  next comment spawns.

## 2026-07-06 — Worktree ops never block the app: background git, pending rows, instant-vanish delete (native app)

Deleting a holibob-sized worktree froze Synth for the whole `rm`; creating one froze it for the
checkout; every `git` call in the app was a synchronous subprocess on the main thread. All worktree
operations now run on a background queue, serialized per repo (concurrent `git worktree` calls race
the repo's locks; different repos run independently), and the UI tells the truth about the interim:

- **Pending rows.** Creating a worktree (⌘K, the create sheet, or the add-workspace picker) adds the
  branch row immediately — grayed, spinner in the shared indicator slot, inert (no sessions can be
  opened in it) — and the row activates in place when the checkout lands. Pending rows are never
  persisted: a quit mid-create must not restore a row whose checkout may never have landed. The
  add-workspace picker closes instantly and materialises its selections as pending rows.
- **Instant-vanish delete.** "Delete worktree" drops the row at once; the folder is atomically
  renamed to a hidden `.deleting-…` sibling (same volume, O(1) regardless of tree size), git's
  admin entry pruned, and the real delete runs afterwards where nobody waits on it. A failed rename
  falls back to the blocking `git worktree remove` — still off the main thread. A launch sweep
  finishes any delete a crash interrupted. Chosen over a lingering grayed row: watching a 30s `rm`
  behind a spinner reads as broken; the row's job is done the moment the user decides.
- **Failures surface as a persistent system toast.** A failed background create/delete drops the
  pending row and raises an error card in the existing deck with the branch · workspace identity —
  no session behind it, so it stays until clicked (⌘↩ reads "dismiss"); unfocused, Notification
  Center is alerted too, and the full git message goes to the log.
- **List perf at hundreds of rows.** The add-workspace branch picker goes lazy and keeps the
  keyboard cursor scrolled into view; `Reveal` no longer builds collapsed subtree content (a
  collapsed workspace of hundreds of branches now costs nothing); row selection is computed by the
  parent and passed down, so a cursor move re-evaluates two rows, not every row; cursor scrolling
  matches working.html's `scrollIntoView({block:'nearest'})` instead of re-centering every press.

Verified against a synthetic repo (300 branches, a 30k-file branch): main-thread probe stays at
single-digit-ms medians through create and delete; the worktree folder vanishes ~230ms after
confirm; the pending → active lifecycle and the failure toast were exercised end-to-end via the
automation socket. Native app only — the pending-row/system-toast design has no working.html
counterpart yet.
## 2026-07-06 — Branch-name inputs turn spaces into dashes as you type (both designs + native app)

Branch names can't contain spaces, but the two places you type a new one — ⌘K's New-worktree
frame and the Create-worktree dialog's Branch name field — accepted them, so "word tree" produced
a name git rejects (and the fuzzy matcher, a strict subsequence, can never match a space against
any branch anyway). Both inputs now rewrite live: each space becomes a dash the instant it's
typed, leading whitespace is dropped, and pasted whitespace runs collapse to a single dash — the
caret stays put through the rewrite.

- **Scoped to branch naming only.** Rename frames and every other input keep their spaces; the
  ⌘K frame opts in via a `dashSpaces` flag (HTML frame field / `PaletteFrame.dashSpaces`), the
  dialog wires the same helper onto its field directly.
- **Rewrite over reject:** the input never shows a name git would refuse, rather than validating
  at submit — you watch the real branch name form as you type, in keeping with inline-as-text
  creates (no error state to design).
