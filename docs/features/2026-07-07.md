# 2026-07-07

## 2026-07-07 — Worktree creation switches optimistically; a slow checkout never yanks the pane (both designs + native app)

Creating a worktree bound the content-pane switch to the wrong event. The pending row appeared at
once, but the pane only moved when the background `git worktree add` resolved — an unpredictable,
variable moment. If the checkout was slow the user had already moved on, so the switch landed as a
delayed, involuntary focus-steal (worst case, moving the keyboard cursor mid-keystroke). An expert
panel (macOS HIG, attention/flow, keyboard power-user, optimistic-UI lenses) converged on one rule:
**a background completion must never move focus on its own — last expressed intent wins.**

- **The switch now rides the create keystroke, not the checkout.** Both `createWorktree` flows
  (⌘K / dialog, new or existing branch) call `openWorktreeSetup(row)` the instant the pending row
  is inserted. The content pane shows a "Setting up worktree…" skeleton (same head shape as a real
  session pane, so the resolve is a cross-fade, not a jump). `openSetupBranchID` is the one new
  piece of state — the pending branch whose skeleton is on screen. Never persisted.
- **Completion resolves in place only while the user is still parked there.**
  `applySessionTemplate` checks `openSetupBranchID == branch.id`: still watching → `open(first)`
  swaps the skeleton for the first template session in place; moved on → the viewport is left
  untouched and the first session gets `unread = true`, announcing the ready worktree with the quiet
  gutter bullet (the same idiom as agent-created browsers) rather than stealing the pane.
- **Any navigation revokes the armed resolve.** Opening a session (`open`) or entering settings
  clears `openSetupBranchID`; so does removing the branch. Create worktree B while A is still
  materialising and A resolves quietly (bullet) while B — now the watched one — resolves in place.
- **Empty template ("start bare") is handled uniformly:** if the user is still watching, the
  skeleton is dropped so the pane settles onto the now-ready (session-less) row instead of hanging
  on a spinner that never resolves — the previous code completed but never switched at all.
- **Failure keeps the established non-jarring path:** a failed `git worktree add` still removes the
  pending row and raises the persistent error toast (`raiseWorktreeError`, bottom-left deck, no
  focus steal); `removeBranch` drops the skeleton so the pane falls back to empty. Inline
  retry-in-pane was left as a separable follow-up.
- The bulk workspace-add path (`confirmAddWorkspace`, importing existing branches) never arms a
  setup skeleton, so it keeps its deliberate no-switch behaviour.

The sidebar pending row now also reads as active (`isActivePill`) while its setup skeleton is the
open pane, tying the grayed row to the pane it drives.

Both design mocks model the same flow: `addBranch` now marks the row `.branch--pending` (grayed +
`.wt-spinner` arc), switches to the `.pane-setup` skeleton via `openWorktreeSetup`/`setupEl`, and on
a simulated ~1.2s `git worktree add` resolves in place only while `setupEl` still points at the row
— otherwise it drops a quiet `.session--unread` bullet. `working.html` and `big-picture-design.html`
stay byte-identical (the diff is still just the `<title>` + the big-picture-only browser/simulator
rows).

## 2026-07-07 — Clicking a terminal link now opens it, routed by host (native app + both designs)

Clicking a link in the terminal did nothing. The terminal is libghostty, which detects the link
under the cursor and hands the host a `GHOSTTY_ACTION_OPEN_URL` action — but `GhosttyApp.handleAction`
only handled child-exit and returned `false` for everything else. Opening a URL is the embedded
apprt's job (libghostty has no macOS fallback for a hosted surface), so the click was silently
dropped. That single unhandled case was the whole bug.

The open question was *where* a clicked link should open: Synth's own in-app browser (a session
Claude can drive over CDP) or the user's real external browser. A panel (head of product, UI/UX,
macOS platform conventions, plus a contrarian briefed to argue external-default) converged on the
tension: the in-app browser is the product's wedge — a clicked `localhost:3000` becoming a session
the agent can inspect and act on is the whole point — but Synth's CEF browser uses **per-session
profile dirs wiped on close** (`CEFEngine`), so it carries no login/cookies/extensions. Routing a
GitHub PR, a Vercel deploy, or an SSO'd dashboard into it means a login wall every time, and the
contrarian's sharpest point stuck: a human's click carries no signal the agent needs the page, so
defaulting everything in-app taxes every click to serve a minority.

### What shipped

- **Route by scheme, then host.** `http`/`https` **and** a loopback host
  (`localhost`/`127.0.0.1`/`0.0.0.0`/`[::1]`/`*.localhost`, mirroring `URL.fromBrowserInput`) → the
  in-app browser. Everything else — real web URLs (which need the user's auth) and every non-web
  scheme (`mailto:`/`tel:`/`file://`/`vscode://`…, which an embedded Chromium renders as a blank or
  error) → `NSWorkspace.shared.open`, the OS default handler. This threads the panel: the one web
  case with no auth problem *and* the maximum agent value (your running dev server) goes in-app;
  the login-bearing web and the non-web schemes go where they belong.
- **One reused browser per source session, not one per click.** A terminal/Claude row remembers its
  link browser (`linkBrowsers`, keyed by source session, dropped on close of either end); reclicking
  a dev-server URL navigates that row instead of spawning a new one — avoids the session-list
  landfill a chatty dev server would otherwise create. Owned by the clicking session when it's a
  Claude row (nests + cascade-closes, extending the ownership model from 2026-07-06), unowned from a
  plain terminal.
- **`OPEN_URL` handler correctness.** The action's `url` is a `char*`+`len`, *not* NUL-terminated —
  decoded by reading exactly `len` bytes (not `String(cString:)`). Empty/whitespace and the `HTML`
  kind (a rendered-content payload, e.g. man pages — not a link) are skipped; `URL(string:)` failure
  falls back to percent-encoding, then drops. Flows onto the bus as `openURLRequested(UUID?, URL)`,
  routed in the store's `apply` exactly like `browserPopupRequested`.
- **Escape hatch surfaced as a button.** The in-app browser toolbar gains an open-in-default-browser
  icon (Phosphor `external`) next to DevTools, alongside the existing ⌘K "Open in default browser"
  it duplicates — a visible one-click path out to the real browser for the loaded page. Landed in
  `working.html`, `big-picture-design.html`, and the native `BrowserBar`.

### Verified

- Build green; app boots clean with the changes.
- Routing decision table checked against 13 representative URLs (localhost/127/0.0.0.0/`[::1]`/
  `*.localhost` → in-app; github/vercel/`notlocalhost.evil.com`/mailto/tel/file/vscode → external),
  all correct — including the security-relevant `notlocalhost.evil.com` *not* matching loopback.
- Not driven end-to-end: a physical libghostty link-click isn't scriptable from the agent harness.
  Manual test: open a terminal, `echo http://localhost:3000` and `echo https://github.com`, cmd+click
  each — the first opens an in-app browser row, the second opens your default browser.

## In-app feedback (⌘⇧F) — one textbox that forks: author spawns a fix, everyone else emails

Feedback on Synth, from inside Synth, with the least possible friction: one textbox, opened from
anywhere with **⌘⇧F** (also a ⌘K "Send feedback…" action and a row in the ⌘? sheet). Enter is a
newline, **⌘↵ sends**, Esc dismisses; an unsent draft persists across reopens. The surface is
identical for everyone — the fork lives only in the outcome, decided silently at send.

The insight that makes it cheap: **this is browser comment-mode pointed at Synth itself.** The
author path reuses the exact machinery that already ships — lazy worktrees, `spawnClaude`, and
CommentMode's boot-and-wait PTY delivery — so the novel code is glue, not new primitives.

- **Two modes, one surface, resolved once at launch (`FeedbackMode`).** Author when git
  `user.email` matches a known author address (or `SYNTH_AUTHOR=1`); everyone else emails
  (`SYNTH_AUTHOR=0` forces it). The user never picks a mode and, until send, never sees a
  difference.
- **Author → a real `feedback/<slug>` worktree with Claude already on it.** `submitFeedback`
  slugifies the first four words, cuts a `feedback/<slug>` branch off the open session's workspace
  (`createWorktree`'s path, with a new `onReady` hook on `materialize` so it spawns a single
  seeded Claude instead of the full session template), and — once the checkout lands and the hook
  seam confirms the Claude live — pastes the feedback + captured context into its PTY. The seed is
  delivered **only to a hook-confirmed-live Claude, never a bare shell** (CommentMode's security
  invariant, carried over verbatim). The session appears quietly; an "On it · feedback/<slug>"
  toast confirms.
- **Everyone else → a pre-filled `mailto:isaac.scarrott11@gmail.com`.** Subject + body built with
  `URLComponents` (so `&`/`+`/newlines encode correctly), capped at ~1600 chars for mailto's
  practical limit. No mail client → the body is copied to the clipboard and a toast says so.
- **Context is captured silently, structural only — never PII.** The author seed carries
  `Here: <session kind> · <branch>`, `State: N workspaces · N working · N unread`, and
  `Env: Synth <version> · macOS <ver> · <theme> theme`. Never file contents, absolute paths,
  terminal output, env values, or the clipboard. The **email body attaches only version/OS** — no
  branch or session names leave the machine. There is no context UI (an earlier "secrets stripped"
  chip was cut for simplicity): the capture is an allowlist of scalar facts, so nothing outside it
  can leak.
- **Design first, then ported.** Landed in `working.html` + `big-picture-design.html` (diff stays
  title + demo rows only), then into the native app: new `FeedbackMode` + feedback methods on
  `AppStore`, `FeedbackSheet` (mirrors `CreateWorktreeSheet`/`DialogFrame`), the ⌘⇧F branch +
  modal-Esc guard entry + overlay in `SynthApp`, the two palette items, the shortcuts row, and
  `GitService.gitUserEmail()`.

Verified: the design drives correctly in a browser (⌘⇧F opens, author send spawns a quiet working
`feedback/*` branch without stealing focus, guest send builds the mailto). The native app builds
clean; the author path was driven headlessly over the control socket (TCC denies synthetic input
on this machine) — it captured structural-only context, ran a real `git worktree add -b
feedback/sidebar-lag-when-i`, spawned a Claude session into it, and raised the "On it" toast; the
mailto URL was confirmed well-formed (`&`→`%26`, newlines/unicode encoded, round-trips). An
independent fidelity audit against the design surfaced two gaps since fixed (the `· N unread`
count, and ASCII-only slugs).

## 2026-07-07 — Author feedback names its own fix; worktree creation is collision-proof (both designs + native app)

The author path of ⌘⇧F previously derived the `feedback/<slug>` branch from the feedback text's
first four alphanumeric words, then ran `git worktree add -b` off that with no guard. Two ways it
hard-failed with **"Couldn't create worktree"**:

- **Collision.** A repeated gripe re-derives the same slug, and the `note` fallback collides on any
  second emoji-only note. `git worktree add -b feedback/<slug>` then fails outright with *"a branch
  named 'feedback/<slug>' already exists"* — no retry, the pending row is dropped, the user sees
  "On it" then an error toast.
- **Too long.** The slug had no length cap, so a single unbroken run (a pasted URL, hash, token —
  non-alphanumerics become spaces, but a long token stays one word) could push the ref component
  past the filesystem's 255-byte limit, and git dies with *"cannot lock ref … File name too
  long"*.

Two changes, both fixing root cause rather than the symptom:

- **Author mode asks for a title.** The sheet now shows a required one-line **Title** field ("Name
  this fix") above the box; the title is the slug, the box is optional detail ("Add detail
  (optional)"). Title autofocuses; Enter moves to the body; ⌘↵ sends; both persist across reopens.
  Send is disabled until the title is non-empty. The seed handed to Claude is `title` + (body if
  any) + captured context. **Email mode is unchanged** — no title field, the one box is required,
  git identity still resolves the fork. The author now controls the branch name instead of
  inheriting a lossy auto-slug.
- **Creation can no longer hard-fail.** The slug is length-capped (six words, 48 chars, trailing
  dashes trimmed) so it can't trip the 255-byte ref limit, and `uniqueFeedbackBranch` resolves
  `feedback/<slug>` against existing branches **and** the planned worktree dir, suffixing `-2`,
  `-3`… on any collision before the row/toast/`git worktree add` ever see the name. A repeated
  gripe now lands a fresh sibling branch instead of erroring.
- **The fix always lands in the Synth repo.** `feedbackWorkspace()` used to target whatever
  workspace was open (or `workspaces.first`) — so a gripe filed while working in an unrelated
  client repo cut the `feedback/*` worktree *there*. It now finds the Synth workspace by its
  source markers (`big-picture-design.html` + `app/Sources/Synth/SynthApp.swift` at the repo root,
  a pairing no other repo has — matches the main checkout or any worktree), falling back to the
  old open/first behaviour only if Synth isn't in the sidebar.

Design first (`working.html` + `big-picture-design.html`, diff stays title + demo rows only —
title input CSS/markup, `feedbackTitle` state, author-forked submit, `slugify` cap, DOM-based
dedupe), then ported to the native app: `feedbackTitle` on `AppStore`, reworked `submitFeedback` /
`startFeedbackFix(title:body:)` / `feedbackSlug` / new `uniqueFeedbackBranch`, and the author-only
`TextField` + focus/placeholder forks in `FeedbackSheet`.

Verified: native app builds clean; the design drives correctly (author send requires a title,
spawns a quiet `feedback/<title-slug>` branch, and a second identical send lands `…-2` instead of
failing).
