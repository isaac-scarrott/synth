## Owned browsers are siblings with a Claude mark, not nested rows (ADR-0011 stage four, revised)

Stage four gave a browser owned by a Claude session a one-indent step under its owner — the indent
alone carried the relation. In use the nesting read as a second level of hierarchy that browsers
don't earn: a browser is still just a session in the branch, not a child of the Claude row. The
extra indent bent the sidebar's flat session list out of shape for a relation that only needed a
tag.

**The relation moves from the indent to the indicator.** An owned browser now sits as a plain
sibling on the shared session indent — same as the Claude row, the terminals, and any unowned
browser. The tie shows instead as a small accent Claude sparkle (terracotta `session__icon--ai`,
12px) in the row's right-hand indicator slot — mirroring the owner's own sparkle icon, so the mark
reads as "belongs to Claude." Browsers are status-less for life (the slot was otherwise empty), so
this reuses the slot rather than adding chrome, and it stays on the sidebar's shared right axis
like every other indicator. The owned row still sits directly beneath its owner (adjacency +
mark), and the tooltip names it: "localhost:5173 · belongs to Claude Code".

Adopt/detach animate the mark, not a margin: adopting swaps the indicator to the accent sparkle;
detaching clears it back to the empty status-less slot. `.session--owned` no longer carries an
indent; `.ind--owned` carries the mark. The comment ladder, cascade-delete, and Move under…/Detach
verbs are unchanged.

Landed in both designs (`working.html`, `big-picture-design.html`) and ported to the native app
(`Sidebar.swift`: `OwnedIndicator`, indent removed, `ownershipHelp` tooltip). Verified by driving a
seeded instance — an owned `localhost:5173` shows the sparkle at the shared indent while an unowned
`docs.stripe.com` shows the empty slot.

## Unread roll-up indicator for a collapsed worktree/branch (both designs + native app)

A collapsed worktree/branch already surfaced what's alive inside it — needs-input (?), error (!),
working, running — via its roll-up indicator. But a session that had simply *finished off-screen
and not yet been seen* (unread, but idle) surfaced nothing: the roll-up fell straight through to
last-activity text ("5h"), so a collapsed row holding output waiting for you looked identical to
one with nothing to show. The one background state the user most wants pulled up to a collapsed
row was the one it dropped.

**The fix — unread joins the roll-up, one rung below liveness.** The precedence is now
needs-input > error > working > running > **unread** > idle. Unread sits *below* the live states
deliberately: while anything in the group is still running or needs attention, that liveness is
the headline and the unread will surface on its own once the group settles. Only when every
session is idle does an unread session get pulled up — in place of the last-activity text, never
over a live signal. Like the other roll-up glyphs it shows **only while collapsed**; expanded, each
session carries its own gutter bullet, so a header roll-up would double-encode.

**Visual — a flat blue dot, the gutter bullet promoted to the indicator axis.** Rendered as a 6px
dot in the shared 16px indicator slot, in the same blue as the row's unread bullet (`--input` /
`Theme.attention`). Flat, no glow — unlike the live liveness dots (running/working, which carry a
colour-matched glow and, for working, a pulse). This sets a clean visual grammar the roll-up now
obeys: **dots = ambient status** (blue = unread), **glyphs = needs your action** (? input, !
error), **glowing/pulsing dots = live** (green run, amber work).

- **Designs.** `working.html` `rollUpGroups` gains the unread fallback (`.udot`, flat `var(--input)`
  dot); the demo's collapsed `fix/sidebar-shadow` worktree gets an unread idle `typecheck` session
  so the indicator is visible. Landed in both `working.html` and `big-picture-design.html` — the
  subset invariant holds (the diff is still only the `<title>` + big-picture's extra
  browser/simulator session rows).
- **Native app.** `Branch.hasUnread` (`Rollup.swift`) + an `UnreadDot` view rendered in
  `BranchRollup`'s idle/none case when collapsed (`Sidebar.swift`). The precedence already lived in
  `RollupState`; unread slots in as the collapsed idle-but-unread fallback, above last-activity.
- **Also fixed en route:** a duplicate top-level `const ICON_EXTERNAL` in `working.html` (introduced
  by the recent ⌘K reconcile/port merges) — two same-scope `const`s are a hard SyntaxError that was
  silently killing the *entire* page script (collapse toggles, ⌘K, roll-ups, everything). Removed
  the redundant second declaration; the script runs clean again.

**Verified.** In the browser, the collapsed `fix/sidebar-shadow` worktree shows the blue unread dot
where its activity text used to be, distinct from the green live dots and consistent with the
left-gutter bullets. In the native app (driven over the control socket on this TCC-restricted
machine): seeded two collapsed branches, created an unread idle browser session in one via the
real `browser.create` path, collapsed it, and captured the window — the branch holding the unread
session shows the blue dot while the sibling branch with nothing unread still shows its "2m"
activity text. `swift build` clean.

## ⌘K root shows one scope, not the whole ancestry (both designs; refines "grouping is scope-aware")

The 2026-07-04 pass split the root frame's context actions into three scope groups —
`Session · <name>`, `Branch · <name>`, `Workspace · <name>` — stacked most-local-first. That read
tidily but still put every ancestor's verbs one arrow-key away: focused on a Claude Code session,
you were a careless Enter from **Remove** on its branch or its workspace. Too many actions, too easy
to misfire on the wrong scope.

This pass narrows the root to a **single scope — the innermost thing you're focused on** — and turns
the parents into navigation rather than adjacent actions:

- **One scope's verbs.** The root frame's context block is now just the focused level: the open
  session's `Session · <name>` (Rename / Delete, plus the browser `Page` group), *or* — when a branch
  row is focused — `Branch · <name>` (New terminal / Claude Code / browser, Rename, Remove), *or* —
  when a workspace head is focused — `Workspace · <name>` (New branch, Settings, Rename, Remove). The
  other two levels no longer appear as actions at all.
- **Parents become "Go to".** The enclosing branch and workspace surface under a new **Go to** group
  as jump rows (right-side chip naming the level), each drilling into that parent's own frame — where
  its full action list already lives. Acting on a parent is now a deliberate one-step-up, not a
  neighbour in the same list. The outermost scope (workspace) shows no Go to.
- **Scoping falls out of `rowLevel(contextRow())`.** `contextActions()` branches on the focused
  row's level and returns `{ items, parents }`; the root frame renders `items` under `grp · unit`
  and `parents` under `Go to`. Search (query typed) is unchanged in shape — the now-scoped context
  actions fold under `Actions`, and the Sessions / Branches / Workspaces groups already let you reach
  anything else.
- **Fixed a merge artifact that had killed the palette.** A reconcile/port commit left a duplicate
  `const ICON_EXTERNAL` (top of the icon block *and* beside `ICON_COPY`), a top-level `SyntaxError`
  that aborted the whole design script — ⌘K wasn't running at all. Removed the second declaration in
  both files.
- **Verified** by driving `working.html`: session focus renders `Session · Claude Code`
  (Rename / Delete) + `Go to` (Branch `feat/command-palette`, Workspace `synth`); branch focus renders
  `Branch · main` + `Go to` (Workspace `synth`); workspace focus renders `Workspace · synth`
  (New branch / Settings / Rename / Remove) with no Go to; selecting a Go to parent drills into its
  frame with that parent's actions + children. No console errors; subset invariant intact.

## 2026-07-08 — Two build channels (Stable / Dev) with distinct identity + a generated app icon

The app had one build path, one bundle id (`tech.holibob.synth`), one Application Support dir, no
icon, and no story for handing a build to a teammate. You couldn't keep a stable Synth installed for
everyday use *and* run a live dev build beside it — they'd fight over LaunchServices, the Dock, and
their saved state. This splits Synth into two channels that install and run side by side.

- **`dev.sh` → "Synth Dev"** (`tech.holibob.synth.dev`, version `0.1-dev+<sha>`): debug build, CEF
  symlinked (instant refresh), launches alongside / `--kill` to replace, amber icon. The live
  development loop.
- **`dist.sh` → "Synth"** (`tech.holibob.synth`, version `0.1`, renamed from `build-app.sh`): release
  build, CEF copied (self-contained), ad-hoc signed, **installed to `/Applications`** and also staged
  at `build/Synth.app` for sharing. The everyday / teammate build.
- **`lib.sh`** holds the one copy of the Info.plist writer and resource stager both channels call, so
  their bundles can't drift (the plist heredoc + resource-copy block used to be duplicated).
- **Coexistence via `AppSupport`.** The seven scattered `applicationSupportDirectory` call sites
  (state, worktrees, browser profiles, instances, comments, browser-mcp, git sweep) now route through
  one `AppSupport.root`, whose folder name is the channel's `CFBundleName` — so "Synth" → `…/Synth`
  and "Synth Dev" → `…/Synth Dev`, fully isolated. `SYNTH_SUPPORT_DIR` overrides the whole sandbox;
  the existing `SYNTH_STATE_DIR` harness override is preserved and still wins for `state.json`.
- **App icon.** A champagne-on-charcoal mark — five rising equalizer/tuning pins flanked by two
  violin-scroll (f-hole) curls, reading as a *synthesis of instruments*. (Earlier Higgsfield gradient
  directions — brushed metal, then a multi-colour "liquid swirl" — were explored and dropped as too
  disconnected from Synth's restrained identity.) Stable keeps the champagne mark; the **Dev variant
  retints only the mark to amber**, deterministically — no AI redraw. `mockicon.swift` does it in one
  pass: keys the mark out of the raw art, composites it at **74%** onto a freshly-drawn charcoal
  squircle (transparent corners, **no rim**), and optionally recolours; `build-icons.sh` packs both
  `.icns`. A single master (`AppIcon-source.png`) + both scripts live in `app/icon/`; the build
  scripts just copy `icon/AppIcon.icns` / `icon/AppIcon-Dev.icns` into the bundle, so swapping the art
  never touches a channel script. (An earlier direct-mask left a faint light rim and a too-small mark
  lost in the Dock — the composite-and-scale approach replaced it.)
- **DEV tag.** The dev build carries a small amber "DEV" pill at the window's top-right, on the
  traffic-light axis opposite the lights — so a dev window is unmistakable at a glance. In both
  designs it's `.dev-tag`, gated by an `is-dev` class on `.app`; in the native app it's `DevTagBadge`
  gated by `isDevChannel` (bundle id ends `.dev`). Absent entirely on stable (no layout shift).
- **Distribution decision (pending impl):** teammates via a private **Homebrew cask** that strips
  quarantine — ad-hoc signing stays, no Apple Developer account needed — with Developer ID
  notarization as a later drop-in behind the same `brew install` command. Also still pending:
  channel-specific CEF helper bundle ids.
- **Verified:** `swift build` clean (one pre-existing future-Swift warning on `controlSocketPath`);
  `dev.sh` launched "Synth Dev" into `…/Synth Dev/` (own `state.json` + `instances/<pid>.json`),
  `dist.sh` built + installed `/Applications/Synth.app` and launched it into `…/Synth/`; both
  Application Support roots present simultaneously, identities and icons confirmed via PlistBuddy.
  DEV tag confirmed rendering top-right in `working.html` over CDP; native `DevTagBadge` compiles and
  is gated on the `.dev` bundle id (live window screenshot blocked only by the terminal's missing
  Screen-Recording permission).
