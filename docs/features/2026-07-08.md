## Owned browsers are siblings with a Claude mark, not nested rows (ADR-0011 stage four, revised)

Stage four gave a browser owned by a Claude session a one-indent step under its owner — the indent
alone carried the relation. In use the nesting read as a second level of hierarchy that browsers
don't earn: a browser is still just a session in the branch, not a child of the Claude row. The
extra indent bent the sidebar's flat session list out of shape for a relation that only needed a
tag.

**The relation moves from the indent to the indicator.** An owned browser now sits as a plain
sibling on the shared session indent — same as the Claude row, the terminals, and any unowned
browser. The tie shows instead as a small accent Claude sparkle (terracotta `session__icon--ai`,
12px) in the row's right-hand indicator slot — mirroring the owner's own sparkle icon, so the mark
reads as "belongs to Claude." Browsers are status-less for life (the slot was otherwise empty), so
this reuses the slot rather than adding chrome, and it stays on the sidebar's shared right axis
like every other indicator. The owned row still sits directly beneath its owner (adjacency +
mark), and the tooltip names it: "localhost:5173 · belongs to Claude Code".

Adopt/detach animate the mark, not a margin: adopting swaps the indicator to the accent sparkle;
detaching clears it back to the empty status-less slot. `.session--owned` no longer carries an
indent; `.ind--owned` carries the mark. The comment ladder, cascade-delete, and Move under…/Detach
verbs are unchanged.

Landed in both designs (`working.html`, `big-picture-design.html`) and ported to the native app
(`Sidebar.swift`: `OwnedIndicator`, indent removed, `ownershipHelp` tooltip). Verified by driving a
seeded instance — an owned `localhost:5173` shows the sparkle at the shared indent while an unowned
`docs.stripe.com` shows the empty slot.

## Unread roll-up indicator for a collapsed worktree/branch (both designs + native app)

A collapsed worktree/branch already surfaced what's alive inside it — needs-input (?), error (!),
working, running — via its roll-up indicator. But a session that had simply *finished off-screen
and not yet been seen* (unread, but idle) surfaced nothing: the roll-up fell straight through to
last-activity text ("5h"), so a collapsed row holding output waiting for you looked identical to
one with nothing to show. The one background state the user most wants pulled up to a collapsed
row was the one it dropped.

**The fix — unread joins the roll-up, one rung below liveness.** The precedence is now
needs-input > error > working > running > **unread** > idle. Unread sits *below* the live states
deliberately: while anything in the group is still running or needs attention, that liveness is
the headline and the unread will surface on its own once the group settles. Only when every
session is idle does an unread session get pulled up — in place of the last-activity text, never
over a live signal. Like the other roll-up glyphs it shows **only while collapsed**; expanded, each
session carries its own gutter bullet, so a header roll-up would double-encode.

**Visual — a flat blue dot, the gutter bullet promoted to the indicator axis.** Rendered as a 6px
dot in the shared 16px indicator slot, in the same blue as the row's unread bullet (`--input` /
`Theme.attention`). Flat, no glow — unlike the live liveness dots (running/working, which carry a
colour-matched glow and, for working, a pulse). This sets a clean visual grammar the roll-up now
obeys: **dots = ambient status** (blue = unread), **glyphs = needs your action** (? input, !
error), **glowing/pulsing dots = live** (green run, amber work).

- **Designs.** `working.html` `rollUpGroups` gains the unread fallback (`.udot`, flat `var(--input)`
  dot); the demo's collapsed `fix/sidebar-shadow` worktree gets an unread idle `typecheck` session
  so the indicator is visible. Landed in both `working.html` and `big-picture-design.html` — the
  subset invariant holds (the diff is still only the `<title>` + big-picture's extra
  browser/simulator session rows).
- **Native app.** `Branch.hasUnread` (`Rollup.swift`) + an `UnreadDot` view rendered in
  `BranchRollup`'s idle/none case when collapsed (`Sidebar.swift`). The precedence already lived in
  `RollupState`; unread slots in as the collapsed idle-but-unread fallback, above last-activity.
- **Also fixed en route:** a duplicate top-level `const ICON_EXTERNAL` in `working.html` (introduced
  by the recent ⌘K reconcile/port merges) — two same-scope `const`s are a hard SyntaxError that was
  silently killing the *entire* page script (collapse toggles, ⌘K, roll-ups, everything). Removed
  the redundant second declaration; the script runs clean again.

**Verified.** In the browser, the collapsed `fix/sidebar-shadow` worktree shows the blue unread dot
where its activity text used to be, distinct from the green live dots and consistent with the
left-gutter bullets. In the native app (driven over the control socket on this TCC-restricted
machine): seeded two collapsed branches, created an unread idle browser session in one via the
real `browser.create` path, collapsed it, and captured the window — the branch holding the unread
session shows the blue dot while the sibling branch with nothing unread still shows its "2m"
activity text. `swift build` clean.
