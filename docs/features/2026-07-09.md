# 2026-07-09

## An agent can close the browsers it opened (ADR-0011 stage two + four, extended)

`browser_create` had no counterpart. A Claude session could open a browser, drive it, finish with
it — and the row stayed on the branch forever, because only the user could delete it. The failure
mode is quiet and cumulative: an agent that checks its own work in a browser leaves a row behind
every time it does, and the sidebar silts up with dead pages nobody asked to keep. The tool surface
was teaching agents to litter.

**`browser_close` closes a browser, and agents are told to use it.** New MCP tool + `browser.close`
control verb, doing exactly what deleting the row does (`store.closeSession`). The description
carries the norm, because a tool that merely *permits* cleanup won't get it: close a browser you
opened only to check your own work; leave it open when you opened it *for* the user — to look at,
or to comment in — and say so. The user's feedback that prompted this named the distinction
precisely: a browser used for testing, where no feedback is wanted, should not survive the turn.

**Ownership is the permission, and it already existed.** Stage four stamps the creating claude as
`ownerSessionID`, and stage four's rule is that ownership changes *only by the user*. So "may I
close this?" is already answered by the tree: a session may close what it owns, nothing else. That
makes the guard fall out for free rather than needing a new permission concept —

- a ⌘K browser is unowned, i.e. the user's, and is refused;
- a browser the user detached, or moved under another claude, is refused (detach/re-parent *is* the
  user reclaiming it — the escape hatch already in the model);
- an external claude (no Synth row, so it owns nothing) is refused, and told why.

*Rejected:* letting any claude close any browser. Stage four's shared surface means any claude may
**drive** any browser — but driving is not destroying, and a second agent deleting the row a first
agent is mid-way through using (or the user is reading) is not a shared surface, it's a race. Read
is shared; lifecycle is owned.

**One extra guard: comment mode is a hard no.** If the browser has comment mode engaged, the user
is picking an element or composing right now, and closing would delete the very thing they were
about to send back to this session. Refused with "the user is commenting in this browser — leave it
open." The check reads `engaged`, not `active`, so it also covers the in-flight CDP attach — the
window where the user has clicked the button but the overlay hasn't landed yet.

Sub-agents share the parent's MCP server and its `SYNTH_SESSION_ID`, so they inherit the owner and
can clean up each other's browsers within the one Claude session — which is right, it's one session.
`sessionId` is required (no "close the focused one"), and closing the focused session clears the
focus pointer.

Verified against a running app with a live CEF engine, driven over the control socket and through
the real MCP server over stdio: the owner closes its browser and the row goes; unowned, other-owned,
external-caller, wrong-kind, and unknown-id closes are each refused with their own message and the
row survives; a close during comment mode is refused, and succeeds once comment mode ends.

## The storefront palette is derived from the app icon (both designs)

The shell had drifted from its own icon. The icon is a pale champagne mark on a charcoal squircle;
the app was neutral achromatic grey with iOS system blue for every selection, focus ring and active
row. Two objects, not one. Sampling `AppIcon-source.png` rather than eyeballing it gave the numbers:
the mark averages `#eedfcc` — `hsl(34, 50%, 87%)`, cream rather than gold — and the squircle runs
`#282b30 → #15181c` holding a steady **hue 223° at ~10% saturation**. The charcoal was never neutral;
that cool slate cast is the thing the app was missing.

**Surfaces are the squircle's own gradient.** `--raised #282b30` is its top stop, `--sidebar #1d1f24`
its middle, `--panel #191b1f` below that, `--canvas #0d0f13` one step past its bottom. The app's
elevation ramp and the icon's gradient are now the same ramp, read in the same direction.

**Champagne is the accent, and it is scarce.** All 19 system-blue call-sites are gone. `--accent`
carries the mark itself (`#eee0cd`) in dark, and spends it on a closed set of attention roles:
selection, focus, the ⌘K active row, the send button, the awaited reply. `--accent-rgb` backs every
alpha wash so a ring's hue flips per theme without its geometry moving.

*Light mode cannot use the mark.* At 87% lightness champagne fails contrast on white, so light
`--accent` is a copper `#a86038` (4.78:1) — same warm family, shifted down the ramp — and a new
`--on-accent` keeps white text off the pale champagne fill.

**The collision we designed around:** `--work` amber sits at hue 38°, four degrees from champagne's
34°. Promoting champagne to accent risked collapsing the "agent working" dot into the brand colour.
So amber stays byte-identical and untouched, the light copper is pulled to ~22° to clear it by 15°,
and no warm status dot is ever born beside it. Blue survives *only* as `--input` (`#3a6db3` /
`#7ea6dc`) — a desaturated sibling of the charcoal's own 223° hue. It means "needs you", never brand.

**Workspace avatar chips were muted onto the palette.** Six saturated stickers (`#6366f1`, `#0ea5e9`,
`#f59e0b`, …) had become the loudest thing on screen, and the chip amber sat 3° from `--work`. They
are now six hues at 34% saturation, each ≥15° from every reserved colour, ≥27° from each other, and
≥4.6:1 for their white letter. Identity survives; the shouting doesn't.

*Rejected:* champagne-only chips (workspaces stop being distinguishable at a glance — identity falls
back to the letter alone), and champagne as the needs-input state (it collides with selection, which
is also champagne, and drags it back toward amber).

**Eight faint greys collapsed into one `--ink-meta`.** `#b0b0b5`, `#b6b6bb`, `#bcbcc1` and friends sat
within 5% lightness of each other — a hierarchy nobody could see, and several failed contrast at
~2.4:1. One token now carries them at 4.63:1. This is the one change that is not a retint.

Colour literals only: 117 lines, no box-shadow offset, radius, border width or easing moved. Verified
by rendering both themes and reading back the computed styles — the ⌘K active row resolves to exactly
`rgba(238, 224, 205, 0.1)`, the mark's own colour.
## One 50pt titlebar band, and the traffic lights moved onto it

The top strip was cramped and its three tenants disagreed about where "the top of the window" was.
The traffic lights sat inside the window's 14pt corner radius; the sidebar's collapse toggle centred
25pt in from the trailing edge while the `+` beneath it centred at 27 and every row indicator at 24,
so nothing lined up; and the sidebar strip (42px in the mock, 44pt in the app) and the pane header
(41.5px / 44pt) were near-misses of each other, putting the lights, the pane title and the DEV tag
on three different centre lines across the seam.

**One token, `--titlebar-h: 50px` / `Theme.titlebarHeight`, and everything hangs off it.** The
sidebar strip and all three pane headers (session, worktree-setup, settings) are that tall by
construction rather than by coincidence, so the band reads as one continuous titlebar. Inside it the
lights take the macOS-standard 20pt corner inset, clear of the radius, centred at y=25. The
collapse/expand toggle grows 26→28pt (17pt glyph, 8pt radius) and lands on the sidebar's control
axis — 24pt in from the trailing edge, shared with the `+` and every row's status indicator. The
`+` moved 3pt to join that axis; it had never been on it. Collapsed, the expand toggle sits at
82pt, a 10pt gap past the lights, and the pane title follows at 122.

**The lights are AppKit's, not ours, and that was the whole problem.** A `.hiddenTitleBar` window
gives a 28pt `NSTitlebarView` with the buttons jammed at x=8, centre y=14. Two approaches were
tried and rejected. An empty *unified* `NSToolbar` makes AppKit grow the titlebar to 52pt and
re-centre the lights for free — but the `NSToolbarView` it installs swallows every click across the
band, so the collapse toggle stops responding. Moving the buttons without growing `NSTitlebarView`
leaves them outside its bounds, where they still draw but no longer hit-test (`hitTest` below the
container returns nil). What works, and is what `WindowChrome.swift` does: grow the titlebar
container to the band height and re-place the buttons *inside* it. `NSTitlebarView` is hit-
transparent except on its own widgets, so our content in the band keeps its clicks and the band
still drags the window. AppKit resets the container on every relayout, so it posts frame-change
notifications back to us and we redo the placement; each write is guarded, so our own resize settles
after one pass. Fullscreen is left alone — there the titlebar is an auto-hiding overlay AppKit owns.

Verified against the real running app, not the mock: the traffic lights report circles at x=20/40/60
(⌀12) with centre y=25 and the container at 50pt; the collapse toggle measures 24.0pt from the
sidebar's trailing edge with centre y=25.00, the same line as the lights; the pane header's hairline
lands at y=49.5. Collapsed, the expand toggle spans 82–110 and the title starts at 122. Hit-testing
from the window's frame view returns `_NSThemeCloseWidget` on the close light and the SwiftUI
hosting view on our toggle, both before and after a real content resize (1708×1016 → 1508×838).

## Synth hosts more than one coding agent: OpenCode joins Claude Code (ADR-0012; both designs + native app)

**What it is.** A session can now run **OpenCode** as well as Claude Code, side by side in the same
branch. Both render as agent rows (the terracotta sparkle), both auto-name themselves from the
agent's own generated conversation title, both drive the liveness/needs-input/error indicators, both
can own a browser and receive click-to-comment context, and both appear as their own "New …" row on
every creation surface (⌘K, the row ⋯ kebab, the new-worktree session template). Settings grows one
flags block per installed agent, each previewing its own binary (`$ claude …` / `$ opencode …`). An
agent that isn't installed never appears anywhere.

**The decision worth remembering: agents are a registry of supervisors, not a second special case.**
`SessionKind.claudeCode` becomes `SessionKind.agent(AgentID)` — which agent a row hosts is *data*.
Nothing in the sidebar, palette, notifications, or persistence switches on a specific agent. The
difference between agents lives in exactly two places: an `AgentDescriptor` (name, binary, install
hints, example flags) and an `AgentSupervisor` — the per-session watcher `CONTEXT.md` already named,
which consumes an agent's raw firehose and emits only derived status facts onto the bus (ADR-0001).
**Adding a third agent is one descriptor plus one supervisor**; no other file changes. That was the
whole point of doing this now rather than bolting OpenCode on beside Claude.

**Why the two supervisors look nothing alike.** Claude Code has no programmatic surface, so Synth
manufactures one (ADR-0008): a PATH shim intercepts `claude`, injects `--settings` carrying hook
commands, and status arrives as `synth-hook event` lines on a unix socket. OpenCode already publishes
what Synth needs — its process serves a typed, ordered `/event` SSE bus where every event carries a
`sessionID`, plus an HTTP API for prompting. So OpenCode's supervisor is a *subscriber*: no hooks, no
injected settings, no socket. The shim survives, dispatching on `argv[0]`, and its only job for
OpenCode is to pass `--port <assigned>` so the server listens where Synth already intends to
subscribe. One server per session, because a server binds to exactly one worktree and a Synth branch
*is* one worktree. Signals generalise from `claude-start`/`claude-end` to `agent-start:<id>` /
`agent-end:<id>`.

**Liveness is asserted by the supervisor, never by the launcher.** This is the subtle one, and it bit
us in verification. Handing a browser comment to an agent that has been *launched* but is not yet
*reachable* drops the text on the floor — and because Claude Code's delivery is a paste followed by
Enter, delivering into the bare shell left by a failed `--resume` would hand a hostile page arbitrary
shell execution. So `.agentReady` is posted by the supervisor alone: Claude Code is ready the moment
its own `SessionStart` hook fires (that hook runs from *inside* the live process, so attaching is
readiness); OpenCode is ready only when its event stream connects, because its shim announces the
launch a beat before the server binds. `liveAgentIDs` — the comment-delivery gate — is written by
`.agentReady` and nothing else. A persisted `.agent` kind is still not liveness, exactly as ADR-0008
insisted.

**Things learned the hard way, all verified against a live OpenCode 1.17.7 rather than its docs:**

- A user **abort** reports as `session.error` with `error.name == "MessageAbortedError"` *and* settles
  the session to idle. It is OpenCode's spelling of Claude's 130/143 — a clean interrupt, never a red
  row. Every `session.error` must be classified by `error.name`.
- **needs-input survives auto-approve.** `permission.asked` is suppressed for auto-approved tools, but
  the `question` channel is permission-independent and still fires. Synth sets
  `OPENCODE_ENABLE_QUESTION_TOOL=1` (the tool is env-gated) so an unattended agent can always stop and
  ask. needs-input = `question.asked` **or** `permission.asked`.
- **Text delivery goes through the TUI prompt API** (`tui/append-prompt` + `tui/submit-prompt`), not
  `session.prompt`: the TUI is the surface the user is watching, and OpenCode creates no session at
  all until its first prompt. Those endpoints *publish an event*, so a 200 only means the server
  accepted it — not that a TUI existed to receive it. Delivery clears the prompt, posts, and re-posts
  until the row actually starts a turn.
- **The server is left unauthenticated.** `OPENCODE_SERVER_PASSWORD` secures it, but OpenCode's own
  TUI races its credentials and dies on the resulting 401, taking the session down. A bare `opencode`
  in any terminal already serves an unauthenticated loopback port, so Synth doesn't widen the exposure
  — it only makes the port predictable rather than random. Inherited `OPENCODE_SERVER_*` is scrubbed
  from spawned sessions for the same reason.
- **The event stream must not use `URLSession.bytes(for:)`.** It buffers, and for a never-ending
  response it intermittently never returns the headers: the socket sits ESTABLISHED and Synth waits
  forever for status it will never see. The stream is a `URLSessionDataDelegate`, requests
  `Accept-Encoding: identity`, and — because the shim announces the launch before the port binds —
  probes `/global/health` before opening the one and only SSE task. A naive 250ms retry loop fires a
  dozen pooled data tasks in the dead window and the one that finally connects never gets a response.

**Persistence and the browser MCP.** `claudeSessionID` becomes `agentSessionID` and `claudeFlags`
becomes per-agent flags; a pre-agents snapshot still decodes (its `"claudeCode"` kind, its lone Claude
flags string, its Claude conversation id all migrate) so nobody's tree or resume is lost. Both agents
get the bundled browser MCP registered per worktree in their own schema: `.mcp.json` (`mcpServers`)
for Claude Code, `opencode.json` (`mcp`, `type: "local"`) for OpenCode — the server reads
`SYNTH_WORKTREE` because OpenCode sets no `CLAUDE_PROJECT_DIR`. OpenCode's MCP tools are named
`synth-browser_<tool>` (underscore), not Claude's `mcp__synth-browser__<tool>`.

**Verified**, not assumed: an 11-check gate driving the built app over its control socket — create →
shim `agent-start` → supervisor readiness → deliver a prompt → working → idle → conversation id
captured → row auto-named — passed 5/5 consecutive cold runs, and Claude Code passed the same
lifecycle (hook-driven) unchanged. Both designs were loaded in a real browser: no JS errors, the
add-session frame offers both agents, and Settings renders a flags block per agent. Subset invariant
held (`big-picture-design.html` is regenerated from `working.html`, so the diff is structurally only
the title plus the extra demo session rows).

## The coding-agent gate: everything the port claimed, driven end to end (`app/harness/agents/`)

**What it is.** Eight suites, 73 checks, driving a real CEF-enabled build over its control socket.
Every claim the entry above makes is now executed rather than asserted: a new worktree's **template**
spawns its OpenCode row and that row goes live; a restored row **resumes** with
`opencode --session <id>` and the agent still recalls the word from before the restart; a background
row raises a **transient done toast** and a **persistent needs-input toast** (and the foreground row
raises none); both agents get their **browser MCP registered** in their own schema and a real OpenCode
server connects it; a **live OpenCode agent drives the embedded CEF browser** through
`browser_list`/`browser_navigate`, and the row renames itself from the new page title; **click-to-comment**
on the live page reaches the owning OpenCode session through the same `__synthComment` binding the
overlay uses; a **user abort** settles the row to idle with no error toast; and Claude Code's hook path
still works. `run.sh` fails fast when the build has no CEF — SwiftPM caches the manifest, so a build
made while `vendor/cef` was absent silently produces a binary with no browser engine and the browser
suites then fail for a reason unrelated to the code under test.

**Two harness seams, because focus and worktrees can't be faked.** `automation.notifRoute` pins a
background transition to the deck or to Notification Center: focus normally decides, and a driven
instance never holds focus on a live desktop (the developer's own apps take it straight back), so
neither branch is otherwise reachable. `automation.createWorktree` is the ⌘K frame's exact call.
Both are `SYNTH_AUTOMATION`-gated, like every other automation verb.

**Claude Code stalls in a brand-new worktree, and it always has.** Claude gates a folder it has never
seen behind *"Yes, I trust this folder"* and fires no `SessionStart` until it is answered — and it is
Synth's own `.mcp.json`, written into every worktree, that makes the dialog appear. So a freshly
created worktree's Claude row sits at that prompt: not live, and therefore not a comment or feedback
target (both ladders poll ~20s for liveness and then give up). One Return and the hook lands in 0.6s.
This is pre-existing, not a consequence of the agent abstraction — OpenCode has no such gate, which is
why the asymmetry only became visible once a second agent existed to compare against. Left as-is
deliberately: the obvious fix is to pre-accept trust for worktrees Synth itself created, and that is a
security decision, not a refactor. (An `enabledMcpjsonServers` injection was tried and **reverted** —
an A/B in a pre-trusted folder showed Claude fires with or without it, so the key was never the fix.
Shipping it would have auto-approved project MCP servers to cure a disease it doesn't treat.)

## Each agent wears its own official mark, and OpenCode is spelled the way it spells itself

**Naming.** The product is **OpenCode**, not "opencode" — `site.webmanifest` declares
`"name": "OpenCode"`, the docs and brand page use it in prose, and the README's logo alt text is
"OpenCode logo". Every user-facing label now says OpenCode; the *command* stays lowercase
(`opencode`, `opencode.json`, `opencode --session <id>`), because that's what it is. `AgentID`'s
raw value stays `"opencode"` too — it's a persisted key, not a label, and renaming it would
invalidate every snapshot.

**Icons.** Agents no longer share one sparkle. `AgentDescriptor` gains a `mark`, and one
`SessionIcon` view is the single place a session's icon is chosen — sidebar rows, pane head, ⌘K rows,
the row ⋯ menu, the notification chip, and an owned browser's indicator, which now **mirrors its
owner's mark** rather than showing a generic sparkle (so the tie names *which* agent owns it).
A third agent that ships no artwork falls back to the sparkle and still looks right.

- **Claude Code → Clawd**, pixel-exact. Anthropic publishes no vector of him, so the authoritative
  source is the sprite `claude` draws on startup: quadrant-block glyphs (`▐▛███▜▌` / `▝▜█████▛▘` /
  `▘▘ ▝▝`) decoded to a 16×5 grid and emitted as ten row-merged rects. Tinted with the agent accent,
  so it adapts to both appearances. **He is 3.2:1**, so in a 14px slot he stands about 4px tall and
  reads as a small mark rather than a legible crab — a consequence of the sprite's own proportions,
  not of the drawing.
- **OpenCode → the official square mark**, the block "o": an outer ring with a two-tone inner block
  whose upper half is left open so the row background shows through. Geometry is the brand's own
  (`opencode.ai/favicon.svg`, viewBox 512, ring 128…384 × 96…416). It uses the **brand's own light/dark
  colour pairs** (ring `#211E1E` / inner `#CFCECD`; inverted to `#F1ECEC` / `#4B4646`) rather than
  Synth's terracotta — the glyph says "agent", the colour doesn't. Only official artwork, as asked.

**A crash found while proving it.** Screenshotting the app meant a client that connected to the
control socket and hung up before reading its reply — and Synth *died*, `rc = -13`. The reply's
`write` raised `SIGPIPE`, whose default action kills the process: **any local process could take the
app down by connecting to the socket and disconnecting.** Fixed by ignoring `SIGPIPE` at process
entry (the standard posture for anything that serves a socket) and setting `SO_NOSIGPIPE` on accepted
control connections; the failed write is now just a failed write. Verified by reproducing the kill,
then watching the patched app survive three hang-ups and still answer a well-behaved client.

## Reverted: ⌘K root shows one scope, not the whole ancestry (2026-07-08)

Backed out `a93d280` in both designs and the native app. The root ⌘K frame again stacks every scope
the context row sits in — `Session · <name>`, `Branch · <name>`, `Workspace · <name>`, most-local
first — and `contextActions()` is back to returning a flat `{ items }` with no `parents` / **Go to**
group. Parents are reachable as actions again, not as jump rows into their own frame.

Two things the revert did *not* restore, because they never belonged to it:

- **The duplicate `const ICON_EXTERNAL` fix stays.** `a93d280` claimed it in its message, but the
  removal had already landed a commit earlier; the SyntaxError that was killing the whole design
  script does not come back.
- **The agent registry stays.** `a93d280` predates `3c61c97` (OpenCode joins Claude Code), so its
  "before" side still hard-coded `ICON_AI` / `addClaude` / "New Claude Code". Restoring that verbatim
  would have silently un-landed OpenCode. The reverted branch group instead keeps
  `AGENTS.forEach(… addAgent(br, k))`, and "Move under …" keeps `agentIcon(data-agent)`.

Net loss worth naming: the workspace scope's **Settings / Rename / Remove** were introduced *by*
`a93d280` and go away with it. The root workspace group is once again just **New branch**; those
verbs remain reachable by drilling into the workspace's own frame.

**Verified.** Both designs' inline scripts parse clean; `swift build` completes; the subset invariant
holds (`diff working.html big-picture-design.html` is still only the `<title>` plus big-picture's
extra browser/simulator session rows).
