# 2026-07-10

## Worktree create trusts the outcome, not the exit code

Feedback: "When I create a new worktree for holibob it errors." Reproduced against the real repo:
`git worktree add` finishes the checkout **first** and only then runs the repo's own
`post-checkout` hook — holibob's husky hook runs `pnpm install --frozen-lockfile`, and a
GUI-launched Synth has no `pnpm` on its PATH (`/usr/bin:/bin:/usr/sbin:/sbin`), so the hook exits
127 and git propagates that as the command's status. The worktree is fully materialised and
registered on the right branch; only the hook failed. Synth trusted the exit code: it dropped the
pending row, raised "Couldn't create worktree… pnpm: command not found", and left an **orphaned
worktree + branch** behind — so the retry failed harder ("branch already exists").

The fix, in `GitService.runWorktreeAdd`: on a non-zero exit, ask git whether the worktree is
actually registered at the planned path on the requested branch (`worktree list --porcelain`,
symlink-resolved path compare). If it is, the create succeeded — the hook's complaint goes to the
log, not the user. If it isn't, the error surfaces exactly as before. The same check absorbs the
orphans older failures left: a retry's "branch already exists" now resolves to the existing
checkout instead of a dead end.

Considered and rejected: disabling repo hooks during `worktree add` (`core.hooksPath=`) — faster
and deterministic, but it silently changes what a checkout means in repos that rely on hooks, and
the per-worktree setup script is not yet native, so there'd be no sanctioned replacement. Giving
the spawned git a login-shell PATH was also rejected: hooks would then block "instant" creation for
minutes (holibob's install runs ~70s) and still fail on any machine-specific tooling gap. The
principle: **Synth's contract is the checkout; the repo's hooks are the repo's business.**

**Verified.** Drove the real binary over the control socket (`SYNTH_AUTOMATION=1`, isolated
`SYNTH_SUPPORT_DIR`/`SYNTH_STATE_DIR`): a repo whose `post-checkout` exits 127 now yields a ready
row, no error toast, worktree on disk and registered. The same script against the pre-fix build
fails exactly as reported — error toast up, row gone, orphan left — so the check catches the bug it
guards against.

## The user-facing taxonomy is settled, and every surface now speaks it (ADR-0013; both designs + native app + storefront)

Synth's language had drifted apart surface by surface. The ⌘K surface named itself three ways in
three places: **"Command palette"** in the ⌘? sheet, **"quick actions"** in the browser home hint,
and **"Search or jump to anything…"** in its own placeholder. Settings told the user about
**"in-app toasts"** while macOS called the same event a notification. A ⌘K row offered to
**"Move under"** an owner whose indentation the 2026-07-08 sibling decision had already removed. None
of it was wrong when it shipped; nothing held the vocabulary, so it drifted.

`CONTEXT.md` is now the glossary and the normative reference. `docs/adr/0013-user-facing-taxonomy.md`
records why the three expensive decisions went the way they did.

- **Workspace becomes Project.** The top row is exactly one git repository. It sat one level above
  the **worktree**, two nouns both opening on `work`, shown as neighbouring rows in the ⌘K delete
  fork ("Remove from sidebar" against "Delete worktree"). One had to go, and worktree cannot: it
  names a real thing on disk that ADR-0007 depends on saying precisely. *Rejected:* **Repository**,
  which is what the row literally is and was close, but "project" is what people say out loud.
- **Command menu.** Covers both halves of what the surface does: it finds things and it does things,
  so "go to X" is a command like any other. *Rejected:* keeping the VS Code term of art, and naming
  it after the keystroke alone (which breaks in the ⌘? sheet, where the right column already *is* the
  keys).
- **Remove / Close / Delete, and red means loss.** Remove drops a row and everything survives. Close
  ends a session and no file dies. Delete destroys the worktree folder and the git branch survives.
  Sessions used to be *deleted*, though deleting one touches no disk at all and the code already
  called it `closeSession`. Renaming it threatened to take the safety signal with it, so red was
  re-based on **recoverability** instead of the filesystem: a busy Close wears red because a turn
  dies with it; an idle Close does not; a Remove never does. The glyph follows the word: trash
  destroys, minus drops a row, × closes. All three verbs used to wear the trash can.
- **Busy.** `running` (green, a process is up) and `working` (amber, an agent mid-turn) were two
  words and two colours for one fact. They merge into one state and one **amber** dot. Amber survived
  because green reads as *healthy*, and a burning agent turn is not a state of health. Merging forced
  a second question nobody had asked: the ex-agent dot still *pulsed* and the ex-process dot sat
  steady, so one state rendered two ways. The pulse is gone (`@keyframes sdot-pulse` deleted). The
  glow already says live, and a dev server left running all afternoon should not breathe at you all
  afternoon.
- **Notification.** One event, whether it lands in Synth's own stacked deck or in Notification
  Center. "Toast" was implementation jargon that had leaked into Settings copy.
- **Belongs to / Attach to / Detach.** A browser opened by an agent belongs to it and sits as a
  sibling wearing its mark. "Move under" promised an indent the sidebar stopped drawing on 2026-07-08.
- **New branch to create, Delete worktree to destroy.** You ask for a branch and the folder is how
  Synth gives it to you; deleting names the folder precisely *because* the git branch survives it.
  The asymmetry is load-bearing. The dialog's "Create worktree" button now says "New branch".
- **Agent is sanctioned user-facing vocabulary.** Say the category when Synth must speak generically
  ("Couldn't start an agent for this comment"); name the product on any control that creates or
  configures one ("New Claude Code", "OpenCode flags"). En route this surfaced a real bug: comment
  mode's chip hardcoded **"New Claude session"** while the code spawns `AgentRegistry.default`, so it
  lied outright whenever the default agent was not Claude. It now reads "New agent session".

Internal symbols keep their old names on purpose. `Palette.swift`, `store.palette`, `SessionKind`,
the `Workspace` type, the `.running`/`.working` enum cases, every CSS class, `data-*` attribute and
persistence key are untouched: renaming them would break persistence, the control socket and the
harness, and buys nothing a glossary entry cannot.

Swept across `working.html`, `big-picture-design.html` (subset invariant held at 20 diff lines), 14
Swift files (`swift build` clean), and the storefront, whose twelve product screenshots were all
re-shot from the swept designs. An adversarial pass over every close path (`d`, the command menu, the
row menu, the control socket) confirmed a busy agent cannot be closed without a confirm through any
of them, and that an idle session owning a browser still warns before taking the browser row with it.
That pass also found the row-menu path is dead code (`store.activeMenu` is never assigned), which is
pre-existing and unrelated.

## Gate-verified: ADR-0013's close semantics, driven against a real build (`app/harness/taxonomy/`)

The taxonomy entry above was landed on a static reading of the code. This drives it. A new suite
seeds a scratch project, launches a real CEF build under `SYNTH_AUTOMATION=1`, spawns real OpenCode
agents, and exercises `store.requestDelete` — the same path the `d` key takes. 17/17 checks pass:

- An **idle** session closes with no dialog at all, and the row is gone.
- A **busy** one (prompted, status `working`) opens a confirm whose crumb reads `Close OpenCode?`,
  never "Delete", and whose confirm row comes back `danger=[True, False]`: the Close is red, Cancel
  never is. Cancelling leaves the session alive.
- An **idle** session that **owns a browser** still confirms, because the browser row dies with it,
  but comes back `danger=[False, False]`. Red marks loss, and nothing here is lost.
- The command menu's root offers `Close`, and no `Delete`, no `New worktree`, no `Move under`.

One seam was added to make the claim observable: `automation.palette` now returns a `danger` array
alongside its labels. Without it, "red iff busy" is invisible from outside the process, and a rule
nobody can observe is a rule nobody can keep.

While there, the amber/champagne worry from 2026-07-09 was measured on real pixels rather than
argued: the busy dot samples exactly `#F5A623`, its hue does sit 6.7° from the champagne accent, but
they differ by 16 L\* and **CIELAB ΔE 65**, so they never read as the same colour. The dot clears
8.13:1 against the sidebar and 4.94:1 on the selection pill. Nothing to fix, and the note is retired.


## Signed, notarized, and self-updating

Synth was shareable only as a zip a teammate had to `xattr -dr com.apple.quarantine` before macOS
would run it. That is the whole distribution story replaced: `dist.sh` now signs with a Developer
ID and the hardened runtime, and `release.sh` notarizes, staples, and publishes a Sparkle appcast
that installed copies check on their own. **Synth > Check for Updates…** is the manual door.

Four things had to be true first, and only one of them was Sparkle.

**The bundle id was `tech.holibob.synth`.** Synth is a personal project that happens to be used at
work; the identifier claimed otherwise, and Notification Center keys its permission grant off it,
so the rename gets vastly cheaper before the first public release than after. It is now
`io.github.isaac-scarrott.synth` — a namespace GitHub demonstrably vouches for, and one that costs
no domain registration. Apple never resolves the string; reverse-DNS is a uniqueness convention,
not an ownership claim. The four CEF helpers now hang off the host channel's id rather than a
hardcoded one, so `Synth Dev`'s helpers stop colliding with the stable build's in Launch Services.

**`CFBundleVersion` was a git short hash.** Sparkle orders releases by it. `a3f9c1` against
`bc05d05` is not an ordering, so the updater would have offered the wrong build or none at all. It
is `git rev-list --count HEAD` now, and the marketing version moved into `app/VERSION`.

**`codesign --deep` cannot sign this bundle.** Apple documents `--deep` as a verification flag, and
it applies one entitlement set to every nested binary. CEF's renderer maps V8's JIT pages and dies
on launch under the hardened runtime without `allow-jit` — an entitlement the browser process has
no business carrying. `sign_app` in `lib.sh` signs inside-out instead: CEF's dylibs, the framework
that vends them, Sparkle's `Autoupdate` and `Updater.app` (standalone Mach-Os the framework seal
never covers), the four helpers with their own entitlements, `synth-hook` (a second executable in
`Contents/MacOS` that a bundle signature seals but never signs), and the app last. No Developer ID
in the keychain still yields an ad-hoc build that runs locally — it just says so, loudly.

**The source repo stays private, so the artifacts cannot live with it.** Sparkle fetches the
appcast and every zip with no credentials, and a private repo's release assets 404 for anyone
unauthenticated — including the updater inside an app the user already installed. A second, public
repo would have worked; a public Tigris bucket on Fly.io (`flyctl storage create --public`) is
better, because object storage gives one flat prefix that serves every version forever. GitHub
scopes assets to the release that carried them, which would have meant rewriting each appcast
enclosure to point at its own tag. Here the URL `generate_appcast` writes is already the URL the
object lives at, and an appcast can name a two-year-old zip and still resolve it.

`RELEASE_BUCKET` and `FEED_URL` live in `lib.sh` because `dist.sh` bakes the feed into every stable
build and `release.sh` publishes to it; if the two ever disagreed, the app would poll an address
nothing publishes to. That is the failure mode with no symptom — every upload succeeds and Sparkle
sees 404 forever — so `release.sh` uploads the binaries first, proves an unauthenticated ranged
`curl` can read the new zip, and only then publishes the appcast that points at it. The `aws` calls
carry your keys and prove nothing about an updater, which carries none. The appcast is the switch
that makes a release live, so it is the last thing written, and the git tag is written after that,
because a tag should record a release that happened.

**A 144MB zip per update is not an update.** CEF is 93% of the bundle and changes only when it is
re-vendored, so `generate_appcast` builds binary deltas against an archive of previous zips kept in
`app/releases/`. The one wrinkle: it stamps a single download prefix onto every item, but each
version's zip is an asset of its own GitHub release. `release.sh` repoints each enclosure at the
tag that actually hosts it, reading the delta filenames off the appcast rather than guessing them
from a naming convention — the EdDSA signature covers the file's bytes, not its URL.

The dev channel never checks for updates. `dev.sh` writes no `SUFeedURL`, and `Updates.swift` reads
the key's absence as "this build does not update itself" — a dev build must not replace itself with
a release build. **Check for Updates…** is then absent rather than greyed out, because a disabled
item promises something that channel never does.
