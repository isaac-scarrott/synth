# 2026-07-12

## ⌘N opens the new-session picker for the branch you're in

`⌘N` now does app-wide what `a` does on a sidebar row: it drops straight into the ⌘K
"New session" frame — terminal, each coding agent, browser — for the branch you're in.
Context resolves exactly like ⌘T's: the focused sidebar row's branch when the keyboard owns
the sidebar, else the open session's branch, else the first available. Where ⌘T commits to
the most common choice in one keystroke, ⌘N is the same intent one step earlier — "new
something here" — without reaching for the sidebar first.

Natively the binding lives in the File menu (`New Session…`), *replacing* the stock
"New Window" ⌘N rather than sitting beside it: Synth is a one-window app, so the default
binding was dead weight on the app's most conventional "make a new thing" chord. Menu
key-equivalents resolve before the first responder sees the event, so it works over a
focused terminal or browser page, same as ⌘T. Both shortcuts sheets (⌘?) list it.

## Synth 0.2.0 — the first release an installed copy can actually take

0.2.0 (build 181) ships ⌘N, the browser MCP server's video recording, background updates, and a
renderer that survives being moved between displays. A minor, not a patch: three of those are new
capability, not repair.

It is also the first release to prove the update path end to end, because 0.1.0 was the first thing
in the bucket and had nothing to be a delta *from*. Now there is: `generate_appcast` diffed 181
against 171 and emitted a **610KB** delta against a 130MB full download — a 99.5% saving, and the
number the `app/releases/` archive exists to produce. That archive is gitignored and lives on one
Mac; if it is ever lost, releases keep succeeding and silently ship full downloads to everyone.

**The update behaviour users get moving to 0.2.0 is 0.1.0's, not 0.2.0's.** Sparkle's install policy
is read from the *running* app, so "update in the background, prompt only to relaunch" — which landed
in this release — cannot govern the update that delivers it. Everyone on 171 gets 171's prompt-first
behaviour one last time. The quiet path only starts being felt on the release *after* this one. This
is inherent to how Sparkle works, not a bug to chase: any change to update behaviour is always one
release late, and that is worth knowing before writing the next one.

Verified as an outsider rather than by exit code: downloaded `Synth.zip` from the public bucket with
no credentials, set the quarantine bit Safari would set, and got `accepted / source=Notarized
Developer ID` from `spctl` with the staple validating offline. Both appcast enclosures — full zip and
delta — carry an `edSignature`; a missing one means installed copies reject the update.
