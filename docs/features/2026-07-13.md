## ‚åòD closes the current context (both designs + native app)

‚åòD now closes "the current context" ‚Äî resolved by the same rule ‚åòK, ‚åòT and ‚åòN already use
(`contextRow()`): the focused sidebar row when the sidebar's keyboard ring is showing, otherwise
the session open in the content pane. From the main window it acts on the current session; from
the sidebar it is exactly `d` ‚Äî same `requestDelete` flow, so an idle unowned session closes
immediately and anything busy, browser-owning, or a branch/workspace drops into the ‚åòK confirm
frame. Not a new close path: one keystroke into the existing one, reachable without first handing
focus to the sidebar (the gap feedback flagged ‚Äî closing the session you're looking at required
‚åò0 first).

Surfaced in the ‚åò? sheet (General ‚Üí "Close current session") and as the key hint on the ‚åòK
Session "Close" action. Guarded like its siblings: no-op in Settings, inert while the palette is
open, plain `d` in the sidebar unchanged, ‚åò‚áßD untouched.

Verified in the browser over CDP: busy open session ‚Üí "Close dev server?" confirm; idle sidebar
row under the ring ‚Üí removed with no prompt; branch row under the ring ‚Üí "Remove
feat/command-menu?" confirm. Subset invariant held (diff = title + big-picture's extra session
rows).

Natively, ‚åòD is a real menu item ‚Äî File > Close Session ‚Äî calling `store.closeContext()`
(Navigation.swift), which mirrors `contextRow()` (`keyboardActive ? cursorRef : openSession`)
into the existing `requestDelete` path; the ‚åò? sheet and the ‚åòK Close item's key hint follow.
Verified against an isolated instance over the control socket (9/9): idle open terminal closed
by ‚åòD with no confirm, cursor fell to the branch row, a second ‚åòD raised "Remove master?",
Esc cancelled with nothing else lost ‚Äî screenshot captured via `automation.screenshot`.

## synth-app MCP server ‚Äî approval-gated worktree creation + handoff

A second bundled MCP server, `synth-app` (`mcp/app-server.mjs`), sibling to `synth-browser`:
where the browser server drives a page, this one drives Synth itself. First (and so far only)
tool: `worktree_create(branch, base?, handoff?)` ‚Äî its description tells agents to call it
PROACTIVELY whenever they're asked to start work that doesn't belong on the current branch,
so unrelated work gets its own branch, checkout and sessions instead of piling onto the one
in flight.

Decisions worth remembering:

- **Two servers, not one.** App control and browser control stay separate processes with
  separate registry entries, because they carry different risk and different defaults: the
  browser server ships ON, the app server ships OFF. A new global Settings section ("MCP
  servers", global scope only ‚Äî native-first like the notification-sounds section) holds the
  two toggles; flipping one re-syncs every managed worktree's `.mcp.json` / `opencode.json`
  immediately, and *disabled means removed* ‚Äî the entry is reconciled out of the configs, so
  agents never even see the tools. Both servers share discovery/control plumbing via
  `mcp/shared.mjs` (extracted from `server.mjs`, no behaviour change).

- **Every mutating verb is approval-gated in the app, not the agent.** `worktree_create`
  sends `app.worktreeCreate` over the control socket; the connection's thread parks on a
  semaphore (never main) while Synth shows a native modal ‚Äî who's asking, workspace chip,
  branch, base, and whether a handoff will follow. Enter creates, Esc / "Not now" / backdrop
  declines. The user's answer is the tool result: approve returns the planned checkout path
  (materialisation continues in the background, failures surface as the usual worktree error
  toast); decline tells the agent to continue on the current branch or ask the user ‚Äî not to
  retry unprompted. The window is 4 minutes app-side inside a 250s socket timeout
  server-side, so the agent always reads a real answer, not a dead pipe. A branch that's
  already a Synth row answers "exists" immediately, prompt-free ‚Äî idempotence over ceremony.

- **Handoff rides the feedback loop's delivery path.** With `handoff` text, the approved
  worktree skips the session template and instead spawns one Claude session seeded with the
  brief ‚Äî `seedFeedbackAgent` generalised to `seedAgent` (mount-for-a-beat PTY boot,
  liveness-gated delivery, same security stance: only a supervisor-confirmed-live agent is
  ever handed text). Without `handoff`, the normal template applies and the work stays with
  the requesting agent.

- **Existing-vs-new branch is decided inside the background git op**, not on the approval
  click's main-thread hop: a branch with a worktree is reused, an existing local/remote
  branch is checked out (`git worktree add` DWIMs remote-tracking names), anything else is
  cut off `base` (repo HEAD when omitted).

- **Headless drivability.** New automation verbs `automation.agentPrompts` /
  `automation.agentPromptResolve` map 1:1 onto the sheet's calls, and harness suite
  `t8_appmcp.py` proves the whole loop end to end: default registration (browser on / app
  absent), refusal while off, per-process enablement via the NSArgumentDomain
  (`-synth-mcp-app '<true/>'` ‚Äî no defaults pollution), prompt ‚Üí approve ‚Üí real checkout +
  seeded Claude row, decline ‚Üí nothing created, "exists" short-circuit, and stale-entry
  removal on relaunch. 17/17 checks pass; `lib.py` gains a `SYNTH_APP` env override so a
  worktree's build can be gated without redirecting other checkouts' harness runs.

## Synth 0.3.0 ‚Äî agents can ask for their own worktree

0.3.0 (build 191) is a minor, not a patch: it carries the two capability changes above. ‚åòD closes
the current context, and the `synth-app` MCP server lets an agent request a worktree behind a native
approval prompt. The second is the reason the version moves ‚Äî Synth now exposes a control surface
that agents drive, so the app is no longer only something you drive by hand.

The app server ships OFF. A user upgrading to 0.3.0 gets the tool installed but disabled, and has to
opt in from Settings > MCP servers. Nothing an agent can do changes until they do, which is the
right default for the first release where an agent can mutate the workspace.

Delta economics held: six deltas, 3.9MB total, against a 132MB full download. Installed copies on
171/181/185 all get a few-MB update. This is the first release where `app/releases/` proved its keep
‚Äî three prior zips on disk meant `generate_appcast` could diff against every build still in the
wild.

Verified as an outsider, not by exit code: anonymous download of the public `Synth.zip`, Safari's
quarantine bit set, `spctl` returning `accepted / source=Notarized Developer ID`, the staple
validating offline, the bundle reporting 0.3.0 / 191, and all nine appcast enclosures (three full
zips + six deltas) carrying an `edSignature`.

Housekeeping the release forced: `app/harness/agents/__pycache__/lib.cpython-314.pyc` was tracked in
git with no ignore rule, so every harness run rewrote it and left the tree dirty ‚Äî which
`release.sh` refuses. Untracked, and `__pycache__/` / `*.pyc` added to `.gitignore`. It would have
blocked every future release the same way.

## synth-app MCP approval moves into ‚åòK (supersedes the 0.3.0 modal)

Feedback on the just-shipped 0.3.0 flow: the agent-worktree approval prompt was a modal sheet, the
one place in the app where an action didn't happen inline as a ‚åòK text frame ‚Äî Palette.swift's own
header comment already states the rule it broke ("create / delete / confirm happen inline as text
‚Äî never a modal"). Same violation `d`-to-remove and ‚åòD fixed for sidebar deletes (ADR-0013) applied
here too, just missed in the MCP server's own PR.

Fixed by routing the prompt through the existing `mode: .confirm` frame machinery instead of a new
one: `PaletteModel.confirmAgentWorktree(_:)` builds a two-item frame ‚Äî "Create `<branch>`" / "Not
now" ‚Äî worded like the sheet's copy (who's asking, base, whether a handoff follows) but as the
placeholder hint rather than laid-out fields. `AppStore.presentAgentPrompt` opens the palette (or
reuses it, discarding whatever frame was showing ‚Äî the same interrupt the modal already did) and
resets its stack to `[root, confirmFrame]`; `presentedAgentPromptID` tracks which prompt currently
owns that frame.

The one behavior worth preserving exactly: a prompt can't just vanish. `closePalette()` now checks
`presentedAgentPromptID` and, if set, calls `resolveAgentPrompt(approved: false)` before clearing ‚Äî
so Esc, ‚åòK, and a backdrop click all decline, same as before. Resolving (from either palette item,
the decline-on-close path, or the automation verb) chains to the next queued prompt via
`advanceAgentPromptPresentation`, reopening the palette on it if the queue isn't empty ‚Äî reproducing
the old reactive-modal behavior (`agentPrompt` = `agentPrompts.first`) without a computed property
driving an overlay.

`AgentWorktreeSheet` (Dialogs.swift) and its `ModalBackdrop` overlay are deleted; `DialogFrame` /
`Field` stay in use by the other sheets (feedback, add-project, new-worktree). No change to the
automation verbs (`automation.agentPrompts` / `automation.agentPromptResolve`) or the MCP server
itself ‚Äî only how the answer gets in front of the user.

## Fix: a hidden, stationary pointer could steal keyboard nav

Feedback: opening ‚åòK sometimes highlighted a row halfway down the list, and the sidebar's
keyboard selection ring could vanish mid-navigation ‚Äî both with the mouse never touching the
trackpad. Root cause: `NSCursor.setHiddenUntilMouseMoves(true)` (fired on every keystroke) hides
the pointer but leaves it parked at its last real position, and SwiftUI's `onHover`/
`onContinuousHover` re-fire from hit-testing whenever a view is laid out under that position ‚Äî
the palette list scrolling under keyboard nav, or the sidebar tree scrolling to the nav cursor,
both count as "hover" with zero actual mouse movement.

Added `AppStore.pointerStale` (`Store.swift`), true from the same keystroke that hides the
pointer until a genuine `mouseMoved` event clears it. Three hover callbacks that mutate real
state off a stale hit-test now check it first: the ‚åòK row hover (`Palette.swift`), the sidebar's
ring-dismiss-on-hover (`Sidebar.swift`), and the notification deck's hover-to-fan
(`NotificationDeck.swift` ‚Äî a phantom hover could pop the deck open uninvited). Left the deck's
remaining purely-cosmetic `.onHover` highlight toggles (RowMenu, BrowserPane, Dialogs,
SettingsPane) alone ‚Äî same latent quirk, but no wrong action results, just a rare mistimed tint,
and gating them needs new store plumbing on several small view structs for little payoff. Matches
working.html, which never had this bug: its hover is a literal `mousemove` DOM listener, not a
hit-test callback that layout alone can trigger.
