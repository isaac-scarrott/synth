# 2026-07-13

## ⌘D closes the current context (both designs + native app)

⌘D now closes "the current context" — resolved by the same rule ⌘K, ⌘T and ⌘N already use
(`contextRow()`): the focused sidebar row when the sidebar's keyboard ring is showing, otherwise
the session open in the content pane. From the main window it acts on the current session; from
the sidebar it is exactly `d` — same `requestDelete` flow, so an idle unowned session closes
immediately and anything busy, browser-owning, or a branch/workspace drops into the ⌘K confirm
frame. Not a new close path: one keystroke into the existing one, reachable without first handing
focus to the sidebar (the gap feedback flagged — closing the session you're looking at required
⌘0 first).

Surfaced in the ⌘? sheet (General → "Close current session") and as the key hint on the ⌘K
Session "Close" action. Guarded like its siblings: no-op in Settings, inert while the palette is
open, plain `d` in the sidebar unchanged, ⌘⇧D untouched.

Verified in the browser over CDP: busy open session → "Close dev server?" confirm; idle sidebar
row under the ring → removed with no prompt; branch row under the ring → "Remove
feat/command-menu?" confirm. Subset invariant held (diff = title + big-picture's extra session
rows).

Natively, ⌘D is a real menu item — File > Close Session — calling `store.closeContext()`
(Navigation.swift), which mirrors `contextRow()` (`keyboardActive ? cursorRef : openSession`)
into the existing `requestDelete` path; the ⌘? sheet and the ⌘K Close item's key hint follow.
Verified against an isolated instance over the control socket (9/9): idle open terminal closed
by ⌘D with no confirm, cursor fell to the branch row, a second ⌘D raised "Remove master?",
Esc cancelled with nothing else lost — screenshot captured via `automation.screenshot`.
