# 2026-07-14

## Device mode: agents drive it too, and rotate stops dressing as reload

Two follow-ups to yesterday's device mode, from feedback.

**The browser MCP server can set device mode** (`browser_device_mode`). Screenshots and clicks
already saw the emulated viewport, but an agent had no way to *enter* it — the one mode of four
it couldn't reach. New control verb `browser.deviceMode` (read with no args; set `on` /
`device` / `landscape`) wrapped by an MCP tool with the focused-session fallback every driving
tool uses. Semantics: naming a device or orientation implies `on` — only an explicit `on: false`
exits; the reply always carries the resulting state plus the fleet, so the no-arg call is the
discovery surface; an unknown device id errors listing valid ids, and the "go to" home errors
("navigate first") exactly where the toolbar button disables. Ownership is deliberately not
checked — stage four's rule: any agent may drive any browser, driving isn't destroying — and the
setters are absolute, not toggles, so an agent stating a mode can't race the user flipping the
same switch. Verified end to end on a real CEF build: MCP stdio → control socket → controller →
CDP, `iphone-se` landing as `innerWidth 667×375 @2` (landscape held from a prior set) and
`on: false` clearing the override — including on a background, never-focused session.

**The rotate button previews, instead of pointing** — the circular `arrow-clockwise` sat one
strip below the toolbar's reload arrow and read as a second refresh (the feedback's exact words).
It's now the device glyph turned to the orientation a press would give: a landscape phone while
portrait, upright while landscape. No new icon: the mock rotates `ICON_DEVICE` with one CSS class
(`browser__btn--turned`), the app passes `rotation:` to the same `BarButton` the reload spin
already uses; `ICON_ROTATE` / `Phosphor.arrowClockwise` are deleted. Both designs (subset
invariant held) + native.
