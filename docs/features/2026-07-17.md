# 2026-07-17

## Session layout & pane splitting — the design, settled in `working.html`

The Synth content surface rendered exactly one open session. This feature turns it into a
splittable **layout**: several sessions on screen at once, side-by-side or stacked, nested
arbitrarily. It was charted and resolved as the [Session layout & pane splitting](../../wayfinder/map.md)
wayfinder effort; the whole map is now closed and the destination is a **handoff brief**, not the
native implementation — that build is the next effort.

**What it is.** You split by dragging a session from the sidebar over the content area — edge
drop-zones (left / right / top / bottom) highlight and dropping subdivides the hovered pane; an outer
rim splits the whole surface; a center zone replaces a pane's session in place. Or you split by
keyboard chord. The result is a **binary pane tree**: each pane hosts exactly one session, there is
always exactly one active pane (a copper ring, shown only inside a split), and there is **no empty
pane** — splits are born filled, unsplit collapses and reflows the sibling. A **360×240 min-pane
floor** is the hard guardrail for both drops and resizes.

**Owned by the branch.** The layout is scoped to the branch — one remembered layout per branch,
persisted and restored on relaunch; the workspace owns none, so workspace-switch reduces to
branch-switch for free. Full-screening a single pane is a **transient, tmux-window-style** view that
stashes the split rather than destroying it — the split stays remembered underneath, and the sidebar
echo keeps showing the band behind the full-screen. The driving example: browser + dev logs
side-by-side while testing, switch to another branch and back, and the split is still there.

**The sidebar mirrors the layout.** A split renders in the sidebar as a bare horizontal band of
session tiles (`.session-group`) — **membership + reading order only, never geometry**: nested trees
flatten to one horizontal ordered band, and past ~3 members the non-active tiles go icon-only
(hover-expands). A tile *is* the session row, so the active accent and the hover-kebab → ⌘K come free.
Dragging a session onto another *in the sidebar* is a second route to the same split.

**Keyboard layer — pure Mac-native chords, no leader.** The "fluid for a tmux user" requirement is met
by dual-binding arrows with `hjkl` aliases (mirroring the sidebar's existing `↑↓`/`jk`), not a modal
prefix — so the layer stays discoverable for everyone and consistent with Synth's `⌘`-chord idiom.
Three arrow-families read as one grammar: **`⌘⇧`+arrow create · `⌘⌥` focus · `⌘⌥⇧` resize**, plus
`⌘⇧⏎` zoom, `⌘⇧U` unsplit, `` ⌘` `` cycle, `⌘1`–`9` focus-by-number. Zoom took `⌘⇧⏎` so
notification-jump `⌘⏎` stayed put; the split block sits before the browser page-verbs so `⌘⌥L` (focus)
wins while bare `⌘L` still reaches the omnibox. A keyboard-created pane is filled by a pick-a-session
`⌘K` frame — the keyboard mirror of the mouse drag-in.

**Execution was in-map** (a deliberate override of wayfinder's plan-don't-do default): the design was
*built*, not just specified. Every shell / interaction / style change landed in **both** `working.html`
and `big-picture-design.html`, so `diff working.html big-picture-design.html` shows only the `<title>`
+ big-picture's extra browser/simulator demo session rows — the subset invariant is the guardrail, and
it is green. The build was verified in a real browser end to end (split → multi-pane render with live
per-pane surfaces, sidebar echo, persistence round-trip across a full reload, container-query header
degradation, ⌘K Unsplit, seam-drag hard-stopping at the floor, every keybinding).

**The deliverable is a handoff brief, not the native app.** The brief
([`wayfinder/assets/008-handoff-brief/handoff.md`](../../wayfinder/assets/008-handoff-brief/handoff.md))
is framed for the **`port-working-html`** skill: `working.html` is the source of truth (each behaviour
cited by stable symbol name), and it carries a slice list mapping 1:1 to the resolved tickets — the
layout-tree **spine** first (it blocks everything), then mouse gestures, resize seams, sidebar echo,
unsplit/close, per-branch persistence, keybindings, and narrow-pane polish fan out. The native work
extends **ADR-0010** (persist & restore across restarts — the per-branch pane tree serializes to disk,
where the mock uses `localStorage`), touches ADR-0009 / 0011 / 0003 / 0013, and should record a new ADR
for the per-branch binary pane tree + sticky full-screen as a new architectural primitive.

**Why these choices.** Recorded across the effort's tickets, in short: an arbitrary nested tree with no
cap (VS Code / tmux feel) over a fixed grid; the browser is *not* special, just a session a terminal
can't render; the sidebar echo shows membership not geometry, because a second geometry to keep in sync
is a second thing to get wrong; the split is sticky because a power user's side-by-side workspace
shouldn't evaporate when they glance at one thing full-screen; and no leader/prefix mode, because
Synth already teaches `⌘`-chords and a mode is a second grammar to learn. The native implementation is
the next effort.

## Browser page zoom (⌘+ / ⌘−)

The embedded browser can now zoom the page from the keyboard, matching every other browser: **⌘+**
zooms in, **⌘−** zooms out (⌘= is accepted as the unshifted twin of ⌘+). Zoom steps a fixed ladder —
`25, 33, 50, 67, 75, 80, 90, 100, 110, 125, 150, 175, 200, 250, 300` % — clamped at both ends, so
each press lands on a clean, familiar stop.

**Why not bare + / −.** The feedback asked for `_` and `+` with no modifier. A loaded page owns the
keyboard once focused, so bare `+`/`−` would be swallowed by a page text field (typing `+` would
zoom instead of inserting a character) and only fire when the page happened to be unfocused —
unreliable in exactly the moment you want it. ⌘-modified keys never reach page text and ride the
same window-wide path as the existing browser verbs (⌘L, ⌘R, ⌘[, ⌘]), so they work over a focused
page and respect the home state (nothing to zoom) for free.

**Reset without stealing ⌘0.** Chrome's reset is ⌘0, but in Synth ⌘0 is a core nav shortcut (focus
sidebar; ⌘1 focuses content) and must stay that. Reset instead lives on a **zoom badge** in the
omnibox row — a compact `125%` pill that appears only off 100% and clicks back to 100%. It doubles as
the live zoom readout, so there's visible feedback on every step and a one-click way home; and since
100% is itself a ladder stop, ⌘− always steps back onto it too. The badge sits at the right end of
the address pill (Chrome's zoom-control spot), left of the comment/device/DevTools cluster.

**Discoverability.** The ⌘K page group gains **Zoom in**, **Zoom out**, and **Reset zoom** (reset
disabled at 100%), and the ⌘? shortcuts sheet's Browser section lists ⌘+ / ⌘− (with ⌘= as the alt).

**Zoom rides navigation.** It re-applies after every paint, so it survives reload and back/forward
within the session — the per-tab feel of a real browser — rather than snapping back to 100% on each
load.

Landed in the design (`working.html` + `big-picture-design.html`, subset invariant intact — the diff
is still just the title + big-picture's extra session rows) and driven end-to-end in the mock: steps,
clamps, badge reset, and reload-persistence all verified. **Follow-up:** port to the native app
(`app/`) — a `setZoomLevel`/`zoomLevel` seam on `BrowserEngine` (CEF `CefBrowserHost::SetZoomLevel`,
WKWebView `pageZoom`), controller zoom state, the ⌘+/⌘− arm in `SynthApp`'s key monitor, and the
badge in `BrowserBar` — via `/port-working-html`.
