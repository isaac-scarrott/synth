# 2026-07-20

## Agent launch lines are arguments, not keystrokes

A new agent row starts as the user's login shell and immediately `exec`s the agent binary. Until now
that line — `exec claude --dangerously-skip-permissions` — was written into the shell's **stdin**
(libghostty's `initial_input`) with a trailing newline standing in for Enter. It is now handed to the
shell as `-c` instead, riding in on `$SYNTH_LAUNCH_COMMAND`, which the login wrapper reads and
unsets before exec'ing `"$SHELL" -l -i -c "$launch"`.

**Why.** Input queued on a tty belongs to whoever reads it first, and that is not necessarily the
shell's line editor. Startup files read the tty themselves all the time: oh-my-zsh's periodic update
prompt takes exactly one keypress (`read -r -k 1`), and version managers and installers ask their own
questions. On a session where that prompt fired, it consumed the queued `e` as its answer and zsh
executed the remainder — `zsh: command not found: xec`. The agent never launched, and a handoff seed
waiting on `agentReady` timed out undelivered.

Nothing about that is zsh-specific, and no amount of waiting fixes it: the theft is a race with
whatever the user's own config chooses to do, so the launch line has to leave stdin entirely. As an
argument it cannot be eaten, on any shell — bash and fish take `-l -i -c` identically. The line still
runs *after* the rc files, so the shim PATH that routes `claude` through `synth-hook` still wins, and
the resume id is still shell-quoted because it is still a word on a command line.

The launch line no longer echoes into the terminal as though it had been typed, which is the one
visible difference: an agent row now opens straight into the agent's TUI.

## Synth's shells don't stop to ask about updates

Sessions Synth opens on the user's behalf — a handoff, a browser comment — have nobody at the
keyboard when the shell starts, so a startup file that stops to ask a question strands the row: the
launch line waits behind a prompt nobody sees, and the seed times out. Every Synth PTY now carries
`DISABLE_AUTO_UPDATE=true`, oh-my-zsh's own switch and the one such prompt that reaches nearly every
Mac. It scopes to Synth's shells only — `omz update` still works, and terminals outside Synth are
untouched. Same reasoning as opencode's `OPENCODE_DISABLE_AUTOUPDATE`: an embedded agent must not
self-update mid-session.

## ⌘K is one command menu: the kebab matches the main window, and negative actions are red

Two inconsistencies in the ⌘K command menu, one fix each.

**One layout, one entry point.** Opening ⌘K from a row's ⋯ kebab (or right-click) used to drill into a
bespoke per-row frame — verbose labels ("Rename dev server…", "Close dev server"), a crumb chip, and
`sec`-divided sections — while ⌘K on the main window showed concise, grouped actions under
"Session · <name>" / "Branch · <name>" / "Project · <name>" headers. Same surface, two personalities.

The kebab now pins the menu's *context* to the clicked row and shows the ordinary grouped root output:
the exact actions ⌘K gives the focused row, only aimed at the row you clicked. The per-row `sessionFrame`
is retired — it existed only for the kebab. The pinned context leads `contextRow()`, so the grouped
actions target the clicked row rather than the focused sidebar row or the open session, which can differ
— the one correctness risk in the change. Fuzzy search by name survives: each action's target name rides
in its `ctx`, which the matcher folds into the searchable text, so a bare "Rename" is still found by its
session's name. Browsing a row's children is covered by the root's Sessions / Branches / Projects search
groups. (The container browse-frames `branchFrame` / `workspaceFrame` stay, since they also carry
scoped browsing and the project rename/remove/settings actions that the grouped root does not.)

**Red is the negative-action signal.** Red used to mean *loss* — a live Close wore it; an idle Close and
every Remove did not (ADR-0013, "red means loss, not disk"). But a destructive verb drawn in the same
colour as a neutral one reads as safe, and people clicked past it. Red now marks the whole
negative-action set, unconditionally: **Close** (always, not only while busy), every **Remove** (branch,
project, "Remove from sidebar"), and **Detach** / **Attach** (the paired containment verbs). **Delete**
stays red. Neutral, additive, and affirmative controls — Cancel, "Not now", a confirm frame's own
affirmative button — stay uncoloured. It is a flag flip (`danger: true`) consumed by the existing danger
styling, not a new view. It supersedes ADR-0013's colour rule in part: that ADR carries a dated note, and
`CONTEXT.md`'s Remove / Close / Red entries carry the new rule.

Landed design-first in `working.html` + `big-picture-design.html` (subset invariant intact), then ported
to the native app (`Palette.swift`).

one open session but a splittable, per-branch-persisted pane tree (drag-to-split drop zones, the

splitting is a headline feature, not a fix.

## A split stays within one branch / worktree — enforced, not just documented

Feature 003 always said "a split stays within one branch", and persistence quietly assumed it —
`serializeLayout` keeps only the owning branch's own sessions, so a cross-worktree pane couldn't even
survive a restart. But nothing stopped one being *built*: every split route pulled from all sessions
across every branch. You could drag a session from branch B onto branch A's surface, pair two rows
from different worktrees, or pick a foreign session in the ⌘⇧+arrow split picker — momentarily
grafting two worktrees into one pane tree until the next save silently dropped it.

Now the invariant is a hard rule at the interaction layer, shared by all three routes via one guard
(`sessionCanJoinLayout`): a session may only join the on-screen layout when it belongs to the branch
that owns the surface. A cross-branch drag reads **red (refuse)** at every rim/edge zone rather than
applying; a cross-branch pair never highlights its target; the split picker lists only the anchor
branch's sessions. A branchless setup skeleton owns no worktree, so nothing may join it. Landed in
`Layout.swift` (`sessionCanJoinLayout`, wired into `resolveDrop` / `performPair` / `pairTarget`) and
`Palette.swift` (split-picker filter).

The spec had the same hole: `working.html` / `big-picture-design.html` carried the "stays within one
branch (003)" comment but every route iterated all sessions across every branch — the app was a
faithful port of an unenforced rule, and the design even kept a `replace` drop-kind (centre-of-pane)
the app had dropped, a fourth way to plant a foreign session. Brought the design in line with the
same guard (`canJoinLayout`, wired into `computeDrop`'s rim/split/replace refuse, `splitFrame`, and
`pairTargetAt`); the working ⊂ big-picture subset invariant holds (diff is still just the title + the
browser/simulator rows).

## Reorder drags mark their drop slot with a copper insertion line

Drag-to-reorder (sidebar tree at every level, and the settings template list) no longer
live-reshuffles the list under the pointer. The list holds still; a copper insertion line — 2px,
rounded, a small dot cap at its left end, `rgba(accent, 0.9)`, painted above the drag ghost — marks
the exact slot the release will land in, and the reorder commits once, on release.

**Why.** The unified session drag already committed on release (it has to: the same gesture resolves
live into reorder / pair / content-split, and a list that reshuffles mid-drag fights the pair and
drop-zone highlights). But commit-on-release with no preview meant there was *no way to see where a
dragged row would land* — the feedback that live reshuffling used to provide had quietly disappeared.
The line restores that signal without bringing the reshuffle back, and the copper ties it to the rest
of the drag language (pair highlight, split drop zones — 010/012).

The line hides when the drop would be a no-op: the faded source row already marks that slot, and a
line beside it would promise a move that isn't one. No-op detection is block-aware — a session that
owns rows (browser under its claude) drags as a block, so "its own slot" spans the block, not just
the row.
