## The sidebar's split band survives switching branches / workspaces

The sidebar echo of a split (012) is per-branch: a branch whose remembered layout binds ≥2
sessions shows its member band even while some *other* branch or workspace is open on the
content surface. The current branch's band keeps echoing the on-screen durable tree; every other
branch's band reads its stashed `Branch.layout` — the same tree a member click restores (014).

**Why.** The layout was already owned by the branch and sticky (014): switching away stashed the
split and clicking any member brought it back intact. But the band — the visible "the split is
still here" — dissolved the moment another branch's layout took the surface, so the sidebar read
as if the split had been destroyed, and the only way to learn otherwise was to click back in.
Feedback (feedback/split-switching-branches-or-workspaces): the band should keep telling the
truth about the branch's remembered arrangement, not about what happens to be on screen.

Consistency follow-through: `inSplit` / `unsplitSession` understand a background branch's
remembered split too, so a band tile's kebab still offers Unsplit and dragging a tile out to the
plain sidebar still leaves the split (013) without having to open the branch first. On-screen
behaviour (including transient zoom) is untouched. App-only for now — `working.html`'s echo still
renders a single band for the current branch; the mock catch-up is a follow-up.

## Split focus is a top-edge bar in the mark's colours

The active pane in a split is marked by a 2px bar across its top edge — charcoal `#1e2126` on
light, champagne `#eee0cd` on dark (the Synth mark's own pair, via a new `--focus` token) — 
sweeping in from the left while the previous pane's bar fades (015). Both ends are inset by
`--radius-app` with soft caps so the bar always clears the shell's rounded corners (top-right;
top-left too once the sidebar is collapsed), keeping every pane's bar identical wherever it sits.

**Why.** Feedback (feedback/changing-focus): the copper ring (004 §4) fenced the whole pane, sat
on top of the content, and read loud for a state that is always true. Ten alternatives were
explored in a temp mock (`focus-ideas.html`) — header tint, dimming, title dot, raised card,
edge spine, corner brackets, surface ring, scrim, live cursor — and the top-edge bar won: one
edge instead of four, zero ink on the content, the browser-tab underline idiom flipped up. The
focus colour gets its own `--focus` token (not `--accent`) so the signal can move independently
of the structural accent; in dark the two coincide on the champagne mark. Superseding note only —
the ring CSS is replaced in both design files; the app port rides with the split-focus click fix
(the pane's SwiftUI tap gesture never fires over the AppKit terminal view, so body clicks don't
activate — first-responder change is the natural trigger).

## Unfocused notifications raise the toast as well as the banner

The in-app toast deck is no longer the frontmost-only surface. A background session's
needs-input / error / done now raises its deck toast whether Synth is focused or not; focus
only gates Notification Center, which posts *on top of* the deck when Synth isn't frontmost
(`routeTransition`: the deck is unconditional for non-open sessions, `toNC` adds the banner).
The open session still never toasts in-app, and still reaches Notification Center unfocused
(open ≠ seen). A done toast's 6-second self-dismiss countdown now only runs while Synth is
frontmost — raised (or still up) unfocused, it waits in the deck and gets its six seconds of
being seen after focus returns before dismissing.

**Why.** Feedback (feedback/toast-and-native-notifications): when Synth isn't focused it
should still display the toast as well as the native notification. A Notification Center
banner slides by while you're in another app and leaves no trace; the deck previously stayed
empty on that path, so returning to Synth showed only the unread bullet. Now the banner
catches the eye in the moment and the toast is standing in the deck when focus returns —
ranked, clickable, ⌘↩-jumpable as ever. `NotifRoute` and the `automation.notifRoute` seam
("deck" / "nc") now pin the *focus rule* (deck-only vs deck + NC) rather than either/or
surfaces. Mock comment-only in both design files (a browser page is always "focused", so the
mock's deck behaviour is unchanged); verified against a driven instance via `automation.key`
⌥⇧N / ⌥⇧D + `automation.notifs` — toast standing with `active:false`, done toast still
standing at t+9s unfocused.
