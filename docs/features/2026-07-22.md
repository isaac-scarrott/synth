# 2026-07-22

## Synth 0.8.0 shipped (build 303)

Minor bump for the new liveness indicator: a working or running session now shows a cyan
5×5 lattice beating outward from its centre, replacing the amber dot that shipped through
0.7.1. Notarized, stapled, and verified credential-less from the public bucket (spctl
accepted / Notarized Developer ID, staple valid, appcast newest `Synth-0.8.0.zip` at
`sparkle:version` 303 with `edSignature` on all 18 enclosures). Users on 0.7.1 take a 697KB
delta rather than the 137MB download.

**Merge note.** `v0.7.1` had been tagged and published from `crash-reporting-mach-port-ordering`
without ever landing on main, so main was missing two shipped crash fixes (the Bluetooth/TCC
abort and Breakpad swallowing crashes) while carrying a *higher* build number. Releasing from
main as-is would have pushed every installed copy onto a build where both crashes returned —
Sparkle orders by `CFBundleVersion`, and 303 > 296, so the regression would have auto-installed.
The tag was merged into main first, making main a superset of everything published; the shipped
bundle carries `NSBluetoothAlwaysUsageDescription`, which is the check that proves it. A release
cut from a branch that never merges back leaves this trap armed for whoever releases next.

## Synth installs from a disk image (0.8.1, build 307)

The download was a zip, which hands someone an app and no place to put it. People ran Synth
straight out of `~/Downloads`, where macOS app translocation executes it read-only from a
randomised mount point — and Synth, which locates its bundled `synth-hook` relative to its own
bundle, cannot find its own tooling there. A first-time installer on another Mac also hit
"Apple could not verify Synth is free of malware", the message Sequoia shows when Gatekeeper
cannot complete its check.

`release.sh` now also builds a `.dmg` holding the app beside an `/Applications` symlink, so the
install is a drag and the app ends up somewhere translocation does not apply. Two properties of
it are load-bearing:

- **It is cut from the already-stapled app, and notarized again in its own right.** A copy
  dragged out of an image carries no trace of the image it came from. Staple only the image and
  the app inside verifies solely while Apple is reachable, failing offline as exactly the
  malware-verification message this set out to fix. Both artifacts therefore carry their own
  ticket, at the cost of a second notarization wait per release.
- **It is built into `build/`, never `app/releases/`.** `generate_appcast` treats every archive
  in that directory as a release enclosure and would have published the image as a second,
  competing update for the same version.

Sparkle's path is untouched: the updater still fetches `Synth-<version>.zip` plus deltas, and
the `Synth.zip` alias stays published because shipped builds reference it. The two artifacts now
have two distinct audiences — people get the image, installed copies get the zip.

Verified as an outsider rather than by exit code: anonymous download of the public `Synth.dmg`
with Safari's quarantine flag set, image assessed `accepted / Notarized Developer ID`, mounted,
and the app *copied out and the image unmounted* before checking it — `codesign --deep --strict`
clean, staple valid, `spctl` accepted standalone, which is the state a real install is in.
Appcast newest `Synth-0.8.1.zip` at `sparkle:version` 307 with `edSignature`; 0.8.0 users take a
295KB delta.

**Site note.** `landing/index.html` is the source, but the deployed site is a separate public
repo (`isaac-scarrott/synth-site`, Pages from root) and nothing automates the copy, so edits here
reached nobody until someone made the hop by hand. The release skill now documents it, including
the ordering: publish the release first, or the download button points at a bucket object that
does not exist yet.

## The release tag names the commit that was built

`release.sh` tagged at the end of its run, reading `HEAD` at that moment. A release spends ~20
minutes waiting on Apple, and a commit made in that window silently moved the tag onto a commit
that was never built — which is what happened to `v0.8.1` on its first push: the app was built
from `Synth 0.8.1` (`git rev-list --count` = 307, the `CFBundleVersion` actually shipped) while
the tag landed on the docs commit made during the wait, counting 308. Rebuilding from the tag
would have produced a build number no release ever had.

The commit is now pinned next to the clean-tree guard, before anything is built, and the tag is
created against that. `v0.8.1` was moved back to the commit it was cut from; the shipped binary
was never in question, since the stray commit touched only the release skill's documentation and
nothing that enters the bundle.
