# 2026-07-23

## Project settings layer on the shared default instead of replacing it

The per-project settings shipped yesterday used a single `Override` switch: off inherited, on
replaced. That fought the actual data. Every real project delta is the shared value *plus*
something — `synth` runs the shared setup then `pnpm install` + `db:generate`, launches `claude`
with the shared `--dangerously-skip-permissions` *plus* `--chrome`, opens the shared Claude *then*
a dev server and a browser. Replace-mode made the common case (append) a manual fork you then had
to maintain, and it hid the default the moment you customised, so "what actually runs here?" had no
on-screen answer.

Now a project holds only its **delta**, shown layered on the shared base in run order:

- **Setup script** — the shared base collapses into a strip ("Shared Synth setup · 4 lines",
  expandable, "Edit in Synth") sitting above the project's own steps, so the stack reads
  top-to-bottom exactly as it executes. A rare project can tick "Don't run the shared setup" to
  replace rather than extend — the one case replace is still reachable, as an explicit choice.
- **Agent flags** — one launch line, `$ claude --dangerously-skip-permissions --chrome`, with the
  shared tokens dim and locked and the project's tail editable inline. A tail flag that repeats a
  shared one strikes the shared token: last-one-wins, rendered, no prose.
- **Sessions** — the shared sessions are locked rows at the top (greyed, tagged `Synth`); the
  project's own are added below, reorderable, removable. The first row overall opens.

Empty delta is pure inheritance — no fork, no dimmed placeholder rows, just the shared layer plus
an empty field inviting additions and a `Clear` that strips the whole delta back to inherited.
This retires the `Override` switch, the `Reset to default` button, and the seed/fork step entirely.

Chosen over three alternatives the user was shown (one unified page, an overview-of-divergence
lead, or polishing the toggle) — the data made layering the honest model. It matches Vercel's
[Environment Variables](https://mobbin.com/screens/4298c799-25f7-41c4-8c8e-021332db9297) "link
shared" additive layer more than its build-settings replace-toggle; the difference is that a Synth
project's values genuinely concatenate.

## The settings redesign is live in the native app

Ported the two working.html settings changes into the SwiftUI app in one coupled pass, since it
all routes through `AppStore`: `SettingsScope` (a sidebar scope list) became `SettingsTab`
(`.app`/`.project`) plus a remembered `settingsProjectID`. `SettingsPane` was rewritten from the
old global-vs-workspace stack into the flat, two-tab, layered surface — sections of right-edge-
control rows on the Synth tab, and on the project tab the shared-setup strip over a delta editor,
the inline `$ claude <shared> <tail>` flag line, and the locked shared sessions above the
project's own.

The sidebar stops swapping to a scope list — the tree stays live under Settings (its `SettingsNav`
/ `ScopeRow` / `BackButton` are gone), so clicking a project row still opens that project, and the
foot Settings button just lights up (`.foot-btn.is-on`). Navigation dropped every settings special
case: `activeRows`/`cursorRef`/`activateCursor` treat Settings like the main screen, and
`toggleExpanded` retargets the project tab at whatever project you touch. `[` / `]` walk the tabs.

The inheritance change is real, not cosmetic: `agentFlags(_:for:)` and `sessionTemplate(for:)` —
both consumed by live agent spawns and worktree creation — moved from override to compose (shared
+ project tail / additions), matching the layered model. The setup script already composed;
`wsSkipScript` adds the rare "run only the project's lines" opt-out.

Verified by driving the built app headless (seeded two projects, opened Settings by keyboard,
in-process screenshots of both tabs): the live tree, lit foot, two tabs, flat sections, and the
full layered project tab — shared strip, `pnpm install`/`db:generate` delta, locked shared Claude
tagged Synth with OPENS above `pnpm dev` + `localhost:3000`, and `$ claude --dangerously-skip-
permissions --chrome` with the shared flag dim and `--chrome` tinted — all render as the HTML does.

## Add project drops the branch picker and the typed path

Adding a project used to be two decisions the user didn't need to make. First a text field
asked for a repository path; then a multi-select sheet ("Add worktrees") listed every branch in
the repo and asked which ones to check out. Both are gone.

**One default branch, no picker.** A freshly added project comes in with exactly its default
branch — the checkout already sitting at the repo root — as its sole worktree row, collapsed.
That branch already has a worktree (the repo folder itself), so it lands as a ready row with no
background checkout. Other branches are added later, one at a time, from the row's existing
"New branch" action. The rationale: the picker forced a branching decision at the moment of
adding a repo, when the near-universal answer is "just the main branch"; deferring it matches how
people actually start. `git worktree list` reports the main worktree first, so the default branch
is `worktrees.first?.branch` — no separate HEAD probe.

**Always the folder picker, never a typed path.** Both entry points — the sidebar `+` and the
⌘K "Add project" action — now open the native macOS folder picker (`NSOpenPanel`). The ⌘K path
previously pushed a "Repository path…" input frame; a user should never be expected to type or
paste a filesystem path when the OS already has a picker for exactly this. `working.html` can't
render native OS chrome, so in the mock a click stands in for "folder chosen" and adds a repo
straight away.

Verified in the running app: driving the ⌘K action opened the OS picker (reached via its
Go-to-folder shortcut), and selecting a repo with three branches (`main`, `feat/alpha`,
`feat/beta`) added a single-branch project whose one row is `main`.
