# 2026-07-23

## Add project drops the branch picker and the typed path

Adding a project used to be two decisions the user didn't need to make. First a text field
asked for a repository path; then a multi-select sheet ("Add worktrees") listed every branch in
the repo and asked which ones to check out. Both are gone.

**One default branch, no picker.** A freshly added project comes in with exactly its default
branch — the checkout already sitting at the repo root — as its sole worktree row, collapsed.
That branch already has a worktree (the repo folder itself), so it lands as a ready row with no
background checkout. Other branches are added later, one at a time, from the row's existing
"New branch" action. The rationale: the picker forced a branching decision at the moment of
adding a repo, when the near-universal answer is "just the main branch"; deferring it matches how
people actually start. `git worktree list` reports the main worktree first, so the default branch
is `worktrees.first?.branch` — no separate HEAD probe.

**Always the folder picker, never a typed path.** Both entry points — the sidebar `+` and the
⌘K "Add project" action — now open the native macOS folder picker (`NSOpenPanel`). The ⌘K path
previously pushed a "Repository path…" input frame; a user should never be expected to type or
paste a filesystem path when the OS already has a picker for exactly this. `working.html` can't
render native OS chrome, so in the mock a click stands in for "folder chosen" and adds a repo
straight away.

Verified in the running app: driving the ⌘K action opened the OS picker (reached via its
Go-to-folder shortcut), and selecting a repo with three branches (`main`, `feat/alpha`,
`feat/beta`) added a single-branch project whose one row is `main`.
